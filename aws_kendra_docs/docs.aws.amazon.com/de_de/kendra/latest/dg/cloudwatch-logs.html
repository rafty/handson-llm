Überwachung von Amazon Kendra mit Amazon CloudWatch Logs

Amazon Kendra verwendet Amazon CloudWatch Logs, um Ihnen einen Einblick in den Betrieb Ihrer Datenquellen zu geben. Amazon Kendra protokolliert Prozessdetails für die Dokumente, während sie indexiert werden. Es protokolliert Fehler aus Ihrer Datenquelle, die während der Indexierung Ihrer Dokumente auftreten. Sie verwenden CloudWatch Logs, um die Protokolldateien zu überwachen, zu speichern und darauf zuzugreifen.

CloudWatch Logs speichert Protokollereignisse in einem Protokollstream, der Teil einer Protokollgruppe ist. Amazon Kendra verwendet diese Funktionen wie folgt:

  • Protokollgruppen — Amazon Kendra speichert all Ihre Log-Streams in einer einzigen Protokollgruppe für jeden Index. Amazon Kendra erstellt die Protokollgruppe, wenn der Index erstellt wird. Die Log-Gruppen-ID beginnt immer mit „aws/kendra/“.

  • Log Stream — Amazon Kendra erstellt für jeden Indexsynchronisierungsauftrag, den Sie ausführen, einen neuen Datenquellen-Log-Stream in der Protokollgruppe. Außerdem wird ein neuer Dokumentprotokollstream erstellt, wenn ein Stream ungefähr 500 Einträge erreicht.

  • Protokolleinträge — Amazon Kendra erstellt bei der Indizierung von Dokumenten einen Protokolleintrag im Log-Stream. Jeder Eintrag enthält Informationen zur Bearbeitung des Dokuments oder zu aufgetretenen Fehlern.

Weitere Informationen zur Verwendung von CloudWatch Protokollen finden Sie unter Was ist Amazon Cloud Watch Logs im Amazon Cloud Watch Logs-Benutzerhandbuch.

Amazon Kendra erstellt zwei Arten von Log-Streams:

  • Datenquellen-Log-Streams

  • Protokolldatenströme

Datenquellen-Log-Streams

Datenquellen-Log-Streams veröffentlichen Einträge zu Ihren Indexsynchronisierungsaufträgen. Jeder Synchronisationsjob erstellt einen neuen Log-Stream, der zum Veröffentlichen von Einträgen verwendet wird. Der Name des Protokollstreams lautet:

data source id/YYYY-MM-DD-HH/data source sync job ID

Für jeden ausgeführten Synchronisationsjob wird ein neuer Log-Stream erstellt.

Es gibt drei Arten von Protokollnachrichten, die in einem Datenquellen-Logstream veröffentlicht werden:

  • Eine Protokollnachricht für ein Dokument, das nicht zur Indexierung gesendet werden konnte. nachfolgend sehen Sie ein Beispiel für diese Nachricht für ein Dokument in einer S3-Datenquelle:

    {
    "DocumentId": "document ID",
    "S3Path": "s3://bucket/prefix/object",
    "Message": "Failed to ingest document via BatchPutDocument.",
    "ErrorCode": "InvalidRequest",
    "ErrorMessage": "No document metadata configuration found for document attribute key  city."
}

  • Eine Protokollnachricht für ein Dokument, das nicht zum Löschen gesendet werden konnte. nachfolgend sehen Sie ein Beispiel für diese Nachricht:

    {
    "DocumentId": "document ID",
    "Message": "Failed to delete document via BatchDeleteDocument.",
    "ErrorCode": "InvalidRequest",
    "ErrorMessage": "Document can't be deleted because it doesn't exist." 
}

  • Eine Protokollmeldung, wenn eine ungültige Metadatendatei für ein Dokument in einem Amazon S3 S3-Bucket gefunden wird. nachfolgend sehen Sie ein Beispiel für diese Nachricht.

    {
    "Message": "Found invalid metadata file bucket/prefix/filename.extension.metadata.json."
}

  • Für SharePoint Datenbankkonnektoren schreibt Amazon Kendra nur dann Nachrichten in den Log-Stream, wenn ein Dokument nicht indexiert werden kann. nachfolgend sehen Sie ein Beispiel für die von Amazon Kendra protokollierte.

    { 
    "DocumentID": "document ID", 
    "IndexID": "index ID", 
    "SourceURI": "", 
    "CrawlStatus": "FAILED", 
    "ErrorCode": "403", 
    "ErrorMessage": "Access Denied", 
    "DataSourceErrorCode": "403"
}

Protokolldatenströme

Amazon Kendra protokolliert Informationen über die Verarbeitung von Dokumenten, während diese indexiert werden. Es protokolliert eine Reihe von Nachrichten für Dokumente, die in einer Amazon S3 S3-Datenquelle gespeichert sind. Es protokolliert Fehler nur für Dokumente, die in einer Microsoft SharePoint - oder einer Datenbankdatenquelle gespeichert sind.

Wenn die Dokumente mithilfe derBatchPutDocument Operation zum Index hinzugefügt wurden, wird der Log-Stream wie folgt benannt:

YYYY-MM-DD-HH/UUID

Wenn die Dokumente mithilfe einer Datenquelle zum Index hinzugefügt wurden, wird der Log-Stream wie folgt benannt:

dataSourceId/YYYY-MM-DD-HH/UUID

Jeder Log-Stream enthält bis zu 500 Nachrichten.

Wenn die Indizierung eines Dokuments fehlschlägt, wird diese Meldung in den Log-Stream ausgegeben:

{
    "DocumentId": "document ID",
    "IndexName": "index name",
    "IndexId": "index ID"
    "SourceURI": "source URI"
    "IndexingStatus": "DocumentFailedToIndex",
    "ErrorCode": "400 | 500",
    "ErrorMessage": "message"
}
