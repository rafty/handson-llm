Wiederholen der vorhergehende Versuch

Das Wiederholungsverhalten umfasst Einstellungen darüber, wie die SDKs versuchen, sich nach Fehlern zu erholen, die auf Anfragen an zurückzuführen sindAWS-Services.

Konfigurieren Sie diese Funktion, indem Sie Folgendes verwenden:

max_attempts- Einstellung für gemeinsam genutzteAWSconfig Dateien
AWS_MAX_ATTEMPTS- Umgebungsvariable

Gibt die maximale Anzahl von Versuchen an, die bei einer Anfrage unternommen werden können.

Standardwert: Wenn dieser Wert nicht angegeben ist, hängt sein Standardwert vom Wert derretry_mode Einstellung ab:

  • Fallsretry_mode jalegacy — Verwendet einen für Ihr SDK spezifischen Standardwert (diemax_attempts Standardeinstellung finden Sie in Ihrem spezifischen SDK-Handbuch oder in der Codebasis Ihres SDK).

  • Fallsretry_mode jastandard — Macht drei Versuche.

  • Fallsretry_mode jaadaptive — Macht drei Versuche.

Gültige Werte: Zahl größer als 0.

retry_mode- Einstellung für gemeinsam genutzteAWSconfig Dateien
AWS_RETRY_MODE- Umgebungsvariable

Gibt an, wie das SDK oder das Entwicklertool versucht, es erneut zu versuchen.

Standardwert:legacy ist die Standardstrategie für Wiederholungen.

Zulässige Werte:

  • legacy— Spezifisch für Ihr SDK (überprüfen Sie Ihr spezifisches SDK-Handbuch oder die Codebasis Ihres SDK).

  • standard— Der Standardsatz von Wiederholungsregeln für alleAWS SDKs. Dieser Modus beinhaltet einen Standardsatz von Fehlern, die wiederholt werden, und die Unterstützung von Wiederholungsquoten. Die standardmäßige maximale Anzahl von Versuchen in diesem Modus beträgt drei, sofern dies nicht ausdrücklich konfiguriertmax_attempts ist.

  • adaptive— Ein experimenteller Wiederholungsmodus, der die Funktionalität des Standardmodus beinhaltet, aber eine automatische clientseitige Drosselung beinhaltet. Da dieser Modus experimentell ist, könnte er das Verhalten in future ändern.

Im Folgenden finden Sie den allgemeinen Pseudocode sowohl für den Modus alsstandard auch für denadaptive Wiederholungsmodus:

MakeSDKRequest() {
  attempts = 0
  loop {
    GetSendToken()
    response = SendHTTPRequest()
    RequestBookkeeping(response)
    if not Retryable(response)
      return response
    attempts += 1
    if attempts >= MAX_ATTEMPTS:
      return response
    if not HasRetryQuota(response)
      return response
    delay = ExponentialBackoff(attempts)
    sleep(delay)
  }
}

Hier sind weitere Details zu den im Pseudocode verwendeten Komponenten:

GetSendToken:

Token-Buckets werden nur imadaptive Wiederholungsmodus verwendet. Token-Buckets erzwingen eine maximale Anforderungsrate, indem ein Token verfügbar sein muss, um eine Anfrage zu initiieren. Der SDK-Client ist so konfigurierbar, dass er die Anfrage entweder schnell ablehnt oder blockiert, bis ein Token verfügbar wird.

Client Side Rate Limiting ist ein Algorithmus, mit dem Anfragen zunächst mit einer beliebigen Rate bis zur Token-Zuweisung gestellt werden können. Nachdem jedoch eine gedrosselte Reaktion erkannt wurde, rate-of-request wird der Client dann entsprechend eingeschränkt. Die Token-Zuteilung wird ebenfalls entsprechend erhöht, wenn erfolgreiche Antworten eingehen.

Mit der adaptiven Ratenbegrenzung können SDKs die Geschwindigkeit, mit der Anfragen gesendet werden, verlangsamen, um der Kapazität von besser gerecht zu werdenAWS-Services.

SendHTTPRequest:

Die meistenAWS SDKs verwenden eine HTTP-Bibliothek, die Verbindungspools verwendet, sodass Sie eine vorhandene Verbindung wiederverwenden können, wenn Sie eine HTTP-Anfrage stellen. Im Allgemeinen werden Verbindungen aufgrund von Drosselungsfehlern wiederverwendet, wenn Anfragen erneut versucht werden. Anfragen werden bei einem erneuten Versuch aufgrund vorübergehender Fehler nicht wiederverwendet.

RequestBookkeeping:

Das Wiederholungskontingent sollte aktualisiert werden, wenn die Anfrage erfolgreich ist. Nur imadaptive Wiederholungsmodusmaxsendrate wird die Statusvariable basierend auf der Art der empfangenen Antwort aktualisiert.

Retryable:

In diesem Schritt wird anhand der folgenden Kriterien bestimmt, ob eine Antwort erneut versucht werden kann:

  • Den HTTP-Statuscode .

  • Der vom Dienst zurückgegebene Fehlercode.

  • Verbindungsfehler, definiert als jeder vom SDK empfangene Fehler, bei dem keine HTTP-Antwort vom Dienst empfangen wird.

Vorübergehende Fehler (HTTP-Statuscodes 400, 408, 500, 502, 503 und 504) und Drosselungsfehler (HTTP-Statuscodes 400, 403, 429, 502, 503 und 509) können potenziell erneut versucht werden. Das SDK-Wiederholungsverhalten wird in Kombination mit Fehlercodes oder anderen Daten aus dem Dienst bestimmt.

MAX_ATTEMPTS:

Wird durch dieconfig Dateieinstellung oder die Umgebungsvariable festgelegt.

HasRetryQuota

In diesem Schritt werden Wiederholungsanforderungen gedrosselt, indem ein Token im Quotenbucket für Wiederholungen verfügbar sein muss. Kontingente für Wiederholungsversuche sind ein Mechanismus, um Wiederholungsversuche zu verhindern, bei denen es unwahrscheinlich ist, dass sie erfolgreich sind. Diese Kontingente sind SDK-abhängig, oft clientabhängig und manchmal sogar von Service-Endpunkten abhängig. Die verfügbaren Kontingenttoken für Wiederholungen werden entfernt, wenn Anfragen aus verschiedenen Gründen fehlschlagen, und bei erfolgreicher Ausführung wieder aufgefüllt. Wenn keine Token mehr vorhanden sind, wird die Wiederholungsschleife beendet.

ExponentialBackoff

Bei einem Fehler, der erneut versucht werden kann, wird die Wiederholungsverzögerung anhand eines verkürzten exponentiellen Backoffs berechnet. Die SDKs verwenden einen verkürzten binären exponentiellen Backoff mit Jitter. Der folgende Algorithmus zeigt, wie die Ruhephase in Sekunden für eine Antwort auf eine Anfrage definiert isti: 

seconds_to_sleep_i = min(b*r^i, MAX_BACKOFF)

Im vorherigen Algorithmus gelten die folgenden Werte:

b = random number within the range of: 0 <= b <= 1

r = 2

MAX_BACKOFF = 20 secondsfür die meisten SDKs. Eine Bestätigung finden Sie in Ihrem spezifischen SDK-Handbuch oder im Quellcode.
