Gambaran umum daftar kontrol akses (ACL)

Daftar kontrol akses (ACL) Amazon S3 memungkinkan Anda mengelola akses ke ember dan objek. Setiap ember dan objek memiliki ACL yang melekat padanya sebagai sumber daya tambahan. Itu menentukan Akun AWS atau grup mana yang diberi akses dan jenis aksesnya. Saat permintaan diterima terhadap sumber daya, Amazon S3 memeriksa ACL yang sesuai untuk memverifikasi bahwa pengirim permintaan memiliki izin akses yang diperlukan.

S3 Object Ownership adalah pengaturan tingkat ember Amazon S3 yang dapat Anda gunakan untuk mengontrol kepemilikan objek yang diunggah ke bucket Anda dan untuk nonaktifkan atau mengaktifkan ACL. Secara default, Kepemilikan Objek diatur ke pengaturan yang diberlakukan pemilik bucket, dan semua ACL dinonaktifkan. Ketika ACL dinonaktifkan, pemilik bucket memiliki semua objek di bucket dan mengelola akses ke mereka secara eksklusif dengan menggunakan kebijakan pengelolaan akses.

Mayoritas kasus penggunaan modern di Amazon S3 tidak lagi memerlukan penggunaan ACL. Kami menyarankan Anda untuk tetap menonaktifkan ACL, kecuali dalam keadaan yang tidak biasa di mana Anda perlu mengontrol akses untuk setiap objek secara individual. Dengan ACL dinonaktifkan, Anda dapat menggunakan kebijakan untuk mengontrol akses ke semua objek di bucket Anda, terlepas dari siapa yang mengunggah objek ke bucket Anda. Untuk informasi selengkapnya, lihat Mengontrol kepemilikan objek dan menonaktifkan ACL untuk bucket Anda.

penting

Jika bucket Anda menggunakan pengaturan yang diberlakukan pemilik bucket untuk Kepemilikan Objek S3, Anda harus menggunakan kebijakan untuk memberikan akses ke bucket dan objek di dalamnya. Dengan pengaturan yang diberlakukan pemilik bucket diaktifkan, permintaan untuk mengatur daftar kontrol akses (ACL) atau memperbarui ACL gagal dan mengembalikan kode kesalahan. AccessControlListNotSupported Permintaan untuk membaca ACL masih didukung.

Saat Anda membuat ember atau objek, Amazon S3 membuat ACL bawaan yang memberikan kendali penuh kepada pemilik sumber daya atas sumber daya tersebut. Hal ini ditunjukkan dalam ACL ember sampel berikut (ACL objek bawaan memiliki struktur yang sama):

<?xml version="1.0" encoding="UTF-8"?>
<AccessControlPolicy xmlns="http://s3.amazonaws.com/doc/2006-03-01/">
  <Owner>
    <ID>*** Owner-Canonical-User-ID ***</ID>
    <DisplayName>owner-display-name</DisplayName>
  </Owner>
  <AccessControlList>
    <Grant>
      <Grantee xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" 
               xsi:type="Canonical User">
        <ID>*** Owner-Canonical-User-ID ***</ID>
        <DisplayName>display-name</DisplayName>
      </Grantee>
      <Permission>FULL_CONTROL</Permission>
    </Grant>
  </AccessControlList>
</AccessControlPolicy>

ACL sampel menyertakan elemen Owner yang mengidentifikasi pemilik dengan ID pengguna kanonik Akun AWS. Untuk petunjuk bagaimana menemukan ID pengguna kanonik Anda, lihat. Menemukan Akun AWS ID pengguna kanonis Elemen Grant mengidentifikasi penerima (baik Akun AWS atau grup yang ditentukan sebelumnya) dan izin yang diberikan. ACL bawaan ini memiliki satu elemen Grant untuk pemilik. Anda memberikan izin dengan menambahkan elemen Grant, dengan setiap pemberian yang mengidentifikasi penerimanya dan izinnya.

catatan

ACL dapat memiliki hingga 100 pemberian.

Siapa itu penerima?

Penerima dapat berupa Akun AWS atau salah satu grup Amazon S3 yang telah ditentukan sebelumnya. Anda memberikan izin kepada Akun AWS dengan menggunakan alamat email atau ID pengguna kanonik. Namun, jika Anda memberikan alamat email pada permintaan pemberian Anda, Amazon S3 akan menemukan ID pengguna kanonik untuk akun tersebut dan menambahkannya ke ACL. ACL yang dihasilkan selalu memuat ID pengguna kanonik untukAkun AWS, bukan alamat email dari. Akun AWS

Saat Anda memberikan hak akses, Anda menentukan setiap penerima sebagai type="value" pasangan, di mana type salah satu dari berikut ini:

  • id— Jika nilai yang disebutkan adalah ID pengguna kanonik dari Akun AWS

  • uri— Jika Anda memberikan izin untuk grup yang telah ditentukan sebelumnya

  • emailAddress— Jika nilai yang disebutkan adalah alamat email dari Akun AWS

penting

Menggunakan alamat email untuk menentukan penerima hanya diberi support di Wilayah AWS berikut ini:

  • Asia Pasifik (Singapura)

Untuk informasi tentang menggunakan akses akun silang, lihat Membuat Peran untuk Mendelegasikan Izin kepada Pengguna IAM dalam Panduan Pengguna IAM.

Menemukan Akun AWS ID pengguna kanonis

ID pengguna kanonik berkaitan dengan Akun AWS Anda. ID ini adalah string panjang karakter, seperti:

79a59df900b949e55d96a1e698fbacedfd6e09d98eacf8f8d5218e7cd47ef2be

Untuk informasi tentang cara menemukan ID pengguna kanonik untuk akun Anda, lihat Menemukan ID pengguna kanonis untuk Akun AWS Anda.

Anda juga dapat mencari ID pengguna kanonik Akun AWS dengan membaca ACL dari bucket atau objek yang diizinkan diakses oleh Akun AWS. Ketika Akun AWS individu diberikan izin berdasarkan permintaan pemberian, entri pemberian ditambahkan ke ACL dengan ID pengguna kanonik akun.

catatan

Jika Anda membuat bucket menjadi publik (tidak disarankan), pengguna yang tidak diautentikasi dapat mengunggah objek ke bucket. Pengguna anonim ini tidak memiliki Akun AWS. Ketika pengguna anonim mengunggah objek ke ember Anda, Amazon S3 akan menambahkan ID pengguna kanonik khusus (65a011a29cdf8ec533ec3d1ccaae921c) sebagai pemilik objek dalam ACL. Untuk informasi selengkapnya, lihat ember Amazon S3 dan kepemilikan objek.

Grup Amazon S3 yang sudah ditentukan sebelumnya

Amazon S3 memiliki serangkaian grup yang telah ditentukan sebelumnya. Saat memberikan akses akun ke sebuah grup, Anda menentukan salah satu dari URI Amazon S3, bukannya ID pengguna kanonik. Amazon S3 menyediakan grup yang telah ditentukan sebelumnya berikut ini:

  • Grup Pengguna yang Diautentikasi – Diwakili oleh http://acs.amazonaws.com/groups/global/AuthenticatedUsers.

    Grup ini mewakili semua Akun AWS. Izin akses ke grup ini Akun AWS memungkinkan setiap mengakses sumber daya. Namun demikian, semua permintaan harus ditandatangani (diautentikasi).

    Awas

    Saat Anda memberikan akses ke grup Pengguna yang Diautentikasi, setiap pengguna yang AWS diautentikasi di dunia dapat mengakses sumber daya Anda.

  • Grup Semua Pengguna – Diwakili oleh http://acs.amazonaws.com/groups/global/AllUsers.

    Izin akses ke grup ini memungkinkan siapa pun di dunia mengakses sumber daya. Permintaan dapat ditandatangani (diautentikasi) atau tidak ditandatangani (anonim). Permintaan yang tidak ditandatangani menghilangkan header Autentikasi dalam permintaan.

    Awas

    Kami sangat merekomendasikan agar Anda tidak pernah memberikan Grup Semua Pengguna izin WRITE, WRITE_ACP, atau FULL_CONTROL. Misalnya, sementara izin WRITE tidak mengizinkan non-pemilik untuk menimpa atau menghapus objek yang ada, izin WRITE masih memungkinkan siapa pun menyimpan objek dalam bucket Anda, yang akan ditagihkan kepada Anda. Untuk perincian lebih lanjut tentang izin ini, lihat bagian berikut Izin apa yang dapat saya berikan?.

  • Grup Log Pengiriman – Diwakili oleh http://acs.amazonaws.com/groups/s3/LogDelivery.

    WRITEizin pada ember memungkinkan grup ini menulis log akses server (lihatPencatatan permintaan menggunakan pencatatan akses server) ke ember.

catatan

Saat menggunakan ACL, penerima dapat berupa Akun AWS atau salah satu grup Amazon S3 yang telah ditentukan sebelumnya. Namun, penerima tidak dapat berupa pengguna IAM. Untuk informasi selengkapnya tentang AWS pengguna dan izin dalam IAM, lihat Menggunakan. AWS Identity and Access Management

Kunci kondisi

Saat Anda memberikan izin kebijakan akses, Anda dapat menggunakan kunci kondisi untuk membatasi nilai ACL pada objek dengan menggunakan kebijakan ember. Kunci konteks berikut sesuai dengan ACL. Anda dapat menggunakan kunci konteks ini untuk mewajibkan penggunaan ACL tertentu dalam permintaan:

  • s3:x-amz-grant-read ‐ Memerlukan akses baca.

  • s3:x-amz-grant-write ‐ Memerlukan akses tulis.

  • s3:x-amz-grant-read-acp ‐ Memerlukan akses baca ke ACL ember.

  • s3:x-amz-grant-write-acp ‐ Memerlukan akses tulis ke ACL ember.

  • s3:x-amz-grant-full-control ‐ Memerlukan kontrol penuh.

  • s3:x-amz-acl ‐ Memerlukan ACL Terekam.

Misalnya kebijakan yang melibatkan header spesifik ACL, lihat. ACL Sampel

ACL sampel pada suatu ember berikut ini mengidentifikasi pemilik sumber daya dan serangkaian pemberian. Formatnya adalah representasi XML dari ACL dalam Amazon S3 REST API. Pemilik ember memiliki FULL_CONTROL sumber daya. Selain itu, ACL menunjukkan bagaimana izin diberikan pada sumber daya kepada dua Akun AWS, yang diidentifikasi oleh ID pengguna kanonik, dan dua grup Amazon S3 yang telah ditentukan sebelumnya yang dibahas pada bagian sebelumnya.

  <AccessControlList>
    <Grant>

      </Grantee>
      <Permission>FULL_CONTROL</Permission>
    </Grant>
    
    <Grant>

ACL Terekam

Amazon S3 men support serangkaian hibah yang telah ditentukan sebelumnya, dikenal sebagai ACLs terekam. Setiap ACL terekam memiliki seperangkat penerima hibah dan izin yang telah ditetapkan. Tabel berikut mencantumkan serangkaian ACL terekam dan pemberian terkait yang telah ditentukan sebelumnya.

ACL Terekam Berlaku untuk Izin ditambahkan ke ACL
private Ember dan objek Pemilik mendapat FULL_CONTROL. Tidak ada orang lain yang memiliki hak mengakses (default).
public-read Ember dan objek Pemilik mendapat FULL_CONTROL. Grup AllUsers (lihat Siapa itu penerima?) mendapat akses READ.
public-read-write Ember dan objek Pemilik mendapat FULL_CONTROL. Grup AllUsers mendapatkan akses READ dan WRITE. Memberikan ini pada ember umumnya tidak disarankan.
aws-exec-read Ember dan objek Pemilik mendapat FULL_CONTROL. Amazon EC2 mendapatkan akses READ ke GET sebuah paket Amazon Machine Image (AMI) dari Amazon S3.
authenticated-read Ember dan objek Pemilik mendapat FULL_CONTROL. Grup AuthenticatedUsers mendapat akses READ.
bucket-owner-read Objek Pemilik objek mendapat FULL_CONTROL. Pemilik bucket mendapat akses READ. Jika Anda menyebutkan ACL terekam ini saat membuat ember, Amazon S3 akan mengabaikannya.
bucket-owner-full-control Objek Pemilik objek dan pemilik ember mendapatkan FULL_CONTROL atas objek. Jika Anda menyebutkan ACL terekam ini saat membuat ember, Amazon S3 akan mengabaikannya.
log-delivery-write Ember Grup LogDelivery mendapatkan izin WRITE dan READ_ACP pada ember. Untuk informasi lebih lanjut tentang log, lihat (Pencatatan permintaan menggunakan pencatatan akses server).

Anda menyebutkan sebuah ACL terekam dalam permintaan Anda dengan menggunakan header x-amz-acl pemintaan. Ketika Amazon S3 menerima permintaan dengan ACL terekam dalam permintaan tersebut, Amazon S3 akan menambahkan pemberian yang telah ditentukan sebelumnya ke ACL dari sumber daya itu.
