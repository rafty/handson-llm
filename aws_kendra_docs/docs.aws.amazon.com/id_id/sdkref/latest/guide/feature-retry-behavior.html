Coba perilaku

Perilaku coba lagi mencakup setelan terkait cara SDK mencoba memulihkan dari kegagalan yang diakibatkan oleh permintaan yang dibuatLayanan AWS.

Konfigurasikan fungsionalitas ini dengan menggunakan yang berikut ini:

max_attempts- PengaturanAWSconfig file bersama
AWS_MAX_ATTEMPTS- Variabel lingkungan

Menentukan jumlah maksimum upaya untuk membuat permintaan.

Nilai default: Jika nilai ini tidak ditentukan, defaultnya tergantung pada nilairetry_mode pengaturan:

  • Jikaretry_mode adalegacy — Menggunakan nilai default khusus untuk SDK Anda (periksa panduan SDK spesifik Anda atau basis kode SDK Anda untukmax_attempts default).

  • Jikaretry_mode adalahstandard - Membuat tiga upaya.

  • Jikaretry_mode adalahadaptive - Membuat tiga upaya.

Nilai yang valid: Jumlah lebih besar dari 0.

retry_mode- PengaturanAWSconfig file bersama
AWS_RETRY_MODE- Variabel lingkungan

Menentukan bagaimana SDK atau alat pengembang mencoba mencoba ulang.

Nilai default:legacy adalah strategi coba ulang default.

Nilai valid:

  • legacy— Khusus untuk SDK Anda (periksa panduan SDK spesifik Anda atau basis kode SDK Anda).

  • standard- Kumpulan aturan percobaan ulang standar di seluruhAWS SDK. Mode ini mencakup serangkaian kesalahan standar yang dicoba ulang, dan dukungan untuk kuota coba lagi. Jumlah maksimum default upaya dengan mode ini adalah tiga, kecualimax_attempts secara eksplisit dikonfigurasi.

  • adaptive- Mode percobaan ulang eksperimental yang mencakup fungsionalitas mode standar tetapi mencakup pelambatan sisi klien otomatis. Karena mode ini eksperimental, mungkin mengubah perilaku di future.

Berikut ini adalah pseudocode tingkat tinggi untuk keduastandard danadaptive coba lagi mode:

MakeSDKRequest() {
  attempts = 0
  loop {
    GetSendToken()
    response = SendHTTPRequest()
    RequestBookkeeping(response)
    if not Retryable(response)
      return response
    attempts += 1
    if attempts >= MAX_ATTEMPTS:
      return response
    if not HasRetryQuota(response)
      return response
    delay = ExponentialBackoff(attempts)
    sleep(delay)
  }
}

Berikut ini adalah rincian lebih lanjut tentang komponen yang digunakan dalam pseudocode:

GetSendToken:

Bucket token hanya digunakan dalam modeadaptive coba lagi. Bucket Token memberlakukan tingkat permintaan maksimum dengan mewajibkan token tersedia untuk memulai permintaan. Klien SDK dapat dikonfigurasi untuk gagal cepat permintaan atau blok sampai token tersedia.

Client Side Rate Limiting adalah algoritma yang awalnya memungkinkan permintaan dibuat pada tingkat berapa pun hingga tunjangan token. Namun, setelah respons yang dibatasi terdeteksi, klien rate-of-request kemudian dibatasi. Tunjangan token juga ditingkatkan sesuai jika tanggapan yang berhasil diterima.

Dengan pembatasan laju adaptif, SDK dapat memperlambat laju pengiriman permintaan agar dapat mengakomodasi kapasitas dengan lebih baikLayanan AWS.

SendHTTPRequest:

Sebagian besarAWS SDK menggunakan pustaka HTTP yang menggunakan kumpulan koneksi sehingga Anda dapat menggunakan kembali koneksi yang ada saat membuat permintaan HTTP. Umumnya, koneksi digunakan kembali saat mencoba ulang permintaan karena kesalahan pembatasan. Permintaan tidak digunakan kembali saat mencoba ulang karena kesalahan sementara.

RequestBookkeeping:

Kuota coba ulang harus diperbarui jika permintaan berhasil. Hanya untuk modeadaptive coba lagi, variabel statusmaxsendrate diperbarui berdasarkan jenis respons yang diterima.

Retryable:

Langkah ini menentukan apakah respons dapat dicoba ulang berdasarkan hal berikut:

  • Kode status HTTP.

  • Kode kesalahan dikembalikan dari layanan.

  • Kesalahan koneksi, didefinisikan sebagai kesalahan yang diterima oleh SDK di mana respons HTTP dari layanan tidak diterima.

Kesalahan sementara (kode status HTTP 400, 408, 500, 502, 503, dan 504) dan kesalahan throttling (kode status HTTP 400, 403, 429, 502, 503, dan 509) semuanya berpotensi dicoba ulang. Perilaku percobaan ulang SDK ditentukan dalam kombinasi dengan kode kesalahan atau data lain dari layanan.

MAX_ATTEMPTS:

Ditentukan oleh pengaturanconfig file atau variabel lingkungan.

HasRetryQuota

Langkah ini membatasi permintaan coba lagi dengan mewajibkan token tersedia di bucket kuota coba lagi. Coba ulang bucket kuota adalah mekanisme untuk mencegah percobaan ulang yang tidak mungkin berhasil. Kuota ini bergantung pada SDK, seringkali bergantung pada klien, dan terkadang bahkan bergantung pada endpoint layanan. Token kuota coba ulang yang tersedia dihapus saat permintaan gagal karena berbagai alasan, dan diisi ulang saat berhasil. Ketika tidak ada token tetap, loop coba lagi keluar.

ExponentialBackoff

Untuk kesalahan yang dapat dicoba ulang, penundaan coba ulang dihitung menggunakan backoff eksponensial terpotong. SDK menggunakan backoff eksponensial biner terpotong dengan jitter. Algoritma berikut menunjukkan bagaimana jumlah waktu untuk tidur, dalam hitungan detik, didefinisikan untuk respons permintaani: 

seconds_to_sleep_i = min(b*r^i, MAX_BACKOFF)

Dalam algoritma sebelumnya, nilai berikut berlaku:

b = random number within the range of: 0 <= b <= 1

r = 2

MAX_BACKOFF = 20 secondsuntuk sebagian besar SDK. Lihat panduan SDK atau kode sumber spesifik Anda untuk konfirmasi.
