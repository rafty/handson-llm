Repetir o comportamento

O comportamento de repetição inclui configurações sobre como os SDKs tentam se recuperar de falhas resultantes de solicitações feitas paraServiços da AWS.

Configure essa funcionalidade usando o seguinte:

max_attempts- configuração deAWSconfig arquivo compartilhado
AWS_MAX_ATTEMPTS- variável de ambiente

Especifica o número máximo de tentativas a serem feitas em uma solicitação.

Valor padrão: Se esse valor não for especificado, seu padrão dependerá do valor daretry_mode configuração:

  • Seretry_mode forlegacy — Usa um valor padrão específico para seu SDK (consulte o guia específico do SDK ou a base de código do SDK para ver pormax_attempts padrão).

  • Seretry_mode forstandard — Faz três tentativas.

  • Seretry_mode foradaptive — Faz três tentativas.

Valores válidos: Número maior que 0.

retry_mode- configuração deAWSconfig arquivo compartilhado
AWS_RETRY_MODE- variável de ambiente

Especifica como o SDK ou a ferramenta do desenvolvedor tentam fazer novas tentativas.

Valor padrão:legacy é a estratégia de repetição padrão.

Valores válidos:

  • legacy— Específico para seu SDK (verifique seu guia específico do SDK ou a base de código do seu SDK).

  • standard— O conjunto padrão de regras de repetição em todosAWS os SDKs. Esse modo inclui um conjunto padrão de erros que são repetidos e suporte para cotas de repetição. O número máximo padrão de tentativas com esse modo é três, a menos quemax_attempts esteja explicitamente configurado.

  • adaptive— Um modo de repetição experimental que inclui a funcionalidade do modo padrão, mas inclui a limitação automática do lado do cliente. Como esse modo é experimental, ele pode mudar o comportamento no future.

A seguir está o pseudocódigo de alto nível para os modosstandard e deadaptive repetição:

MakeSDKRequest() {
  attempts = 0
  loop {
    GetSendToken()
    response = SendHTTPRequest()
    RequestBookkeeping(response)
    if not Retryable(response)
      return response
    attempts += 1
    if attempts >= MAX_ATTEMPTS:
      return response
    if not HasRetryQuota(response)
      return response
    delay = ExponentialBackoff(attempts)
    sleep(delay)
  }
}

Aqui estão mais detalhes sobre os componentes usados no pseudocódigo:

GetSendToken:

Os buckets de tokens são usados somente no modoadaptive de repetição. Os buckets de tokens impõem uma taxa máxima de solicitação ao exigir que um token esteja disponível para iniciar uma solicitação. O cliente SDK pode ser configurado para falhar rapidamente na solicitação ou bloquear até que um token seja disponibilizado.

O Client Side Rate Limiting é um algoritmo que inicialmente permite que as solicitações sejam feitas de qualquer forma, até o limite do token. No entanto, depois que uma resposta limitada é detectada, o cliente rate-of-request é então limitado adequadamente. O subsídio simbólico também é aumentado adequadamente se respostas bem-sucedidas forem recebidas.

Com a limitação de taxa adaptativa, os SDKs podem diminuir a taxa na qual as solicitações são enviadas para melhor acomodar a capacidade deServiços da AWS.

SendHTTPRequest:

A maioria dosAWS SDKs usa uma biblioteca HTTP que usa grupos de conexões para que você possa reutilizar uma conexão existente ao fazer uma solicitação HTTP. Geralmente, as conexões são reutilizadas ao repetir solicitações devido a erros de limitação. As solicitações não são reutilizadas ao tentar novamente devido a erros transitórios.

RequestBookkeeping:

A cota de novas tentativas deve ser atualizada se a solicitação for bem-sucedida. Somente para oadaptive modo de repetição, a variável de estadomaxsendrate é atualizada com base no tipo de resposta recebida.

Retryable:

Essa etapa determina se uma resposta pode ser repetida com base no seguinte:

  • Código de status do HTTP .

  • O código de erro retornado pelo serviço.

  • Erros de conexão, definidos como qualquer erro recebido pelo SDK no qual uma resposta HTTP do serviço não é recebida.

Erros transitórios (códigos de status HTTP 400, 408, 500, 502, 503 e 504) e erros de limitação (códigos de status HTTP 400, 403, 429, 502, 503 e 509) podem potencialmente ser repetidos. O comportamento de repetição do SDK é determinado em combinação com códigos de erro ou outros dados do serviço.

MAX_ATTEMPTS:

Especificado pela configuraçãoconfig do arquivo ou pela variável de ambiente.

HasRetryQuota

Essa etapa simplifica as solicitações de repetição ao exigir que um token esteja disponível no intervalo de cotas de repetição. Os intervalos de cotas de repetição são um mecanismo para evitar novas tentativas que provavelmente não serão bem-sucedidas. Essas cotas dependem do SDK, geralmente dependem do cliente e às vezes até dependem dos endpoints do serviço. Os tokens de cota de repetição disponíveis são removidos quando as solicitações falham por vários motivos e reabastecidos quando são bem-sucedidas. Quando não restam tokens, o loop de repetição é encerrado.

ExponentialBackoff

Para um erro que pode ser repetido, o atraso da nova tentativa é calculado usando o recuo exponencial truncado. Os SDKs usam retrocesso exponencial binário truncado com instabilidade. O algoritmo a seguir mostra como a quantidade de tempo para dormir, em segundos, é definida para uma resposta à solicitaçãoi: 

seconds_to_sleep_i = min(b*r^i, MAX_BACKOFF)

No algoritmo anterior, os seguintes valores se aplicam:

b = random number within the range of: 0 <= b <= 1

r = 2

MAX_BACKOFF = 20 secondspara a maioria dos SDKs. Consulte o guia específico do SDK ou o código-fonte para confirmação.
