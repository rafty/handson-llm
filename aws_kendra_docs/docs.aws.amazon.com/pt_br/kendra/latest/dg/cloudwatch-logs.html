Monitorando o Amazon Kendra com o Amazon CloudWatch Logs

O Amazon Kendra usa o Amazon CloudWatch Logs para fornecer informações sobre a operação de suas fontes de dados. O Amazon Kendra registra os detalhes do processo dos documentos à medida que eles são indexados. Ele registra erros de sua fonte de dados que ocorrem enquanto seus documentos estão sendo indexados. Você usa CloudWatch registros para monitorar, armazenar e acessar os arquivos de log.

CloudWatch Os registros armazenam eventos de log em um fluxo de registros que faz parte de um grupo de registros. O Amazon Kendra usa esses recursos da seguinte forma:

  • Grupos de registros — O Amazon Kendra armazena todos os seus fluxos de registros em um único grupo de registros para cada índice. O Amazon Kendra cria o grupo de registros quando o índice é criado. O identificador do grupo de registros sempre começa com “aws/kendra/”.

  • Fluxo de registros — O Amazon Kendra cria um novo fluxo de log da fonte de dados no grupo de registros para cada trabalho de sincronização de índice que você executa. Ele também cria um novo fluxo de registro de documentos quando um fluxo atinge aproximadamente 500 entradas.

  • Entradas de registro — O Amazon Kendra cria uma entrada de registro no fluxo de registros à medida que indexa documentos. Cada entrada fornece informações sobre o processamento do documento ou sobre quaisquer erros encontrados.

Para obter mais informações sobre o uso de CloudWatch registros, consulte O que são registros do Amazon Cloud Watch no Guia do usuário do Amazon Cloud Watch Logs.

O Amazon Kendra cria dois tipos de fluxos de log:

  • Streams de log da fonte de dados

  • Strefluxos de log de documentos

Streams de log da fonte de dados

Os fluxos de log da fonte de dados publicam entradas sobre seus trabalhos de sincronização de índices. Cada trabalho de sincronização cria um novo fluxo de log que ele usa para publicar entradas. O nome do stream de log é:

data source id/YYYY-MM-DD-HH/data source sync job ID

Um novo fluxo de log é criado para cada execução do trabalho de sincronização.

Há três tipos de mensagens de log publicadas em um stream de log da fonte de dados:

  • Uma mensagem de registro para um documento que não foi enviado para indexação. Veja a seguir um exemplo dessa mensagem para um documento em uma fonte de dados do S3:

    {
    "DocumentId": "document ID",
    "S3Path": "s3://bucket/prefix/object",
    "Message": "Failed to ingest document via BatchPutDocument.",
    "ErrorCode": "InvalidRequest",
    "ErrorMessage": "No document metadata configuration found for document attribute key  city."
}

  • Uma mensagem de registro de um documento que não foi enviado para exclusão. Veja a seguir um exemplo dessa mensagem:

    {
    "DocumentId": "document ID",
    "Message": "Failed to delete document via BatchDeleteDocument.",
    "ErrorCode": "InvalidRequest",
    "ErrorMessage": "Document can't be deleted because it doesn't exist." 
}

  • Uma mensagem de log quando um arquivo de metadados inválido para um documento em um bucket do Amazon S3 é encontrado. Veja a seguir um exemplo dessa mensagem.

    {
    "Message": "Found invalid metadata file bucket/prefix/filename.extension.metadata.json."
}

  • Para conectores de banco de dados SharePoint e de banco de dados, o Amazon Kendra só grava mensagens no fluxo de log se um documento não puder ser indexado. Veja a seguir um exemplo da mensagem de erro registrada pelo Amazon Kendra.

    { 
    "DocumentID": "document ID", 
    "IndexID": "index ID", 
    "SourceURI": "", 
    "CrawlStatus": "FAILED", 
    "ErrorCode": "403", 
    "ErrorMessage": "Access Denied", 
    "DataSourceErrorCode": "403"
}

Strefluxos de log de documentos

O Amazon Kendra registra informações sobre o processamento de documentos enquanto eles estão sendo indexados. Ele registra um conjunto de mensagens para documentos armazenados em uma fonte de dados do Amazon S3. Ele registra erros somente em documentos armazenados em uma fonte de dados da Microsoft SharePoint ou de um banco de dados.

Se os documentos foram adicionados ao índice usando aBatchPutDocument operação, o fluxo de log será nomeado da seguinte forma:

YYYY-MM-DD-HH/UUID

Se os documentos foram adicionados ao índice usando uma fonte de dados, o fluxo de log será nomeado da seguinte forma:

dataSourceId/YYYY-MM-DD-HH/UUID

Cada fluxo de log contém até 500 mensagens.

Se a indexação de um documento falhar, essa mensagem será enviada para o fluxo de log:

{
    "DocumentId": "document ID",
    "IndexName": "index name",
    "IndexId": "index ID"
    "SourceURI": "source URI"
    "IndexingStatus": "DocumentFailedToIndex",
    "ErrorCode": "400 | 500",
    "ErrorMessage": "message"
}
