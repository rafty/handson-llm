将自定义同义词库添加到索引

要向索引添加自定义同义词，请在同义词库文件中指定它们。在Amazon Kendra使用同义词时，可以包含特定业务或专业术语。通用英语同义词（例如leader, head）已内置Amazon Kendra于同义词库文件中，不应包含在同义词库文件中。 Amazon Kendra支持所有响应类型的同义词，包括DOCUMENT响应类型和/QUESTION_ANSWER 或ANSWER响应类型。 Amazon Kendra目前不支持添加标记为停用词的同义词。此内容将包含在future 版本中。

Amazon Kendra在同义词之间建立关联。例如，使用同义词对Dynamo, Amazon DynamoDB，将 Dynamo 与Amazon Kendra相关联Amazon DynamoDB。查询 “什么是发电机？” 然后返回诸如 “什么是Amazon DynamoDB？” 之类的文档。使用同义词，Amazon Kendra可以更轻松地找出相关性。

同义词库文件是存储在存储Amazon S3桶中的文本文件。请参阅将同义词库添加到索引。

同义词库文件使用 S olr 同义词格式。 Amazon Kendra对每个索引的同义词库库的数量有限制。参见配额。

同义词在以下情景中可能很有用：

  • 不是传统英语同义词的专业术语，例如NLP, Natural Language Processing。

  • 具有复杂语义关联的专有名词。例如，在机器学习中，这些是公众不太可能理解的名词cost, loss, model performance。

  • 例如，不同形式的产品名称Elastic Compute Cloud, EC2。

  • 特定领域或特定业务的术语，例如产品名称。例如，Route53, DNS。

以下情形请勿使用同义词：

  • 通用英语同义词，例如leader, head. 这些同义词不是特定域的，在这些场景中使用同义词可能会产生意想不到的效果。

  • 排印错误，例如teh => the。

  • 形态变体，例如名词的复数和所有格、形容词的比较和最高级形式以及动词的过去时、过去分词和渐进形式。比较和最高级形容词的一个例子是good, better, best.

  • Unigram（单个单词）停用词，例如WHO。同义词库中不允许使用 Unigram 停用词，并且不包括在搜索范围内。例如，WHO => World Health Organization被拒绝。W.H.O.但是，您可以将停用词用作同义词，也可以将停用词用作多词同义词的一部分。例如，of不允许但United States of America已接受。

自定义同义词可以扩展查询范围以涵盖特定业务的同义词，从而轻松提高Amazon Kendra对业务特定术语的理解。尽管同义词可以提高搜索准确性，但重要的是要了解同义词如何影响延迟，这样才能为此进行优化。

同义词的一般规则是：查询中与同义词匹配和扩展的术语越多，对延迟的潜在影响就越大。影响延迟的其他因素包括索引文档的平均大小、索引的大小、对搜索结果的任何筛选以及Amazon Kendra索引的总体负载。不匹配任何同义词的查询不受影响。

关于同义词如何影响延迟的一般准则：

使用案例 延迟增加*
典型的自然语言或关键字查询，每个 3 到 5 个单词 小于 15%
1 个查询术语扩展为 3 个同义词
索引大约 500,000 个文档（平均每个文档有 10.48 KB 的提取文本）或 30,000 个常见问题解答/问题对

* 性能因您在索引上对同义词和配置的具体使用而异。最好测试搜索性能，以便为您的特定用例获得更准确的基准。

如果您的同义词库很大，术语扩展率很高，并且您的延迟增加不在可接受的范围内，则可以尝试以下一种或两种方法：

  • 修剪同义词库以降低扩展率（每个术语的同义词数量）。

  • 缩小术语的总体覆盖范围（同义词库中的行数）。

或者，您可以增加配置容量（虚拟存储单元）以抵消延迟的增加。
