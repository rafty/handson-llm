Amazon S3metadati del documento

È possibile aggiungere metadati, informazioni aggiuntive su un documento, ai documenti in unAmazon S3 bucket utilizzando un file di metadati. Ogni file di metadati è associato a un documento indicizzato.

I file di metadati devono essere archiviati nello stesso bucket dei file indicizzati. È possibile specificare una posizione all'interno del bucket per i file di metadati utilizzando la console o ilS3Prefix campo delDocumentsMetadataConfiguration parametro quando si crea un'origineAmazon S3 dati. Se non si specifica unAmazon S3 prefisso, i file di metadati devono essere archiviati nella stessa posizione dei documenti indicizzati.

Se si specifica unAmazon S3 prefisso per i file di metadati, questi si trovano in una struttura di directory parallel ai documenti indicizzati. Amazon Kendracerca i metadati solo nella directory specificata. Se i metadati non vengono letti, verifica che la posizione della directory corrisponda alla posizione dei metadati.

Gli esempi seguenti mostrano come la posizione del documento indicizzato è mappata alla posizione del file di metadati. Nota che laAmazon S3 chiave del documento viene aggiunta alAmazon S3 prefisso dei metadati e quindi viene aggiunto il suffisso.metadata.json per formare ilAmazon S3 percorso del file di metadati. LaAmazon S3 chiave combinata, con ilAmazon S3 prefisso e il.metadata.json suffisso dei metadati, non deve contenere più di un totale di 1024 caratteri. Si consiglia di mantenere laAmazon S3 chiave al di sotto dei 1000 caratteri per tenere conto dei caratteri aggiuntivi quando si combina la chiave con il prefisso e il suffisso.

Bucket name:
     s3://bucketName
Document path:
     documents
Metadata path:
     none
File mapping
     s3://bucketName/documents/file.txt -> 
        s3://bucketName/documents/file.txt.metadata.json

     s3://bucketName

I metadati del documento sono definiti in un file JSON. Il file deve essere un file di testo UTF-8 senza un indicatore BOM. Il nome del file JSON deve essere<document>.<extension>.metadata.json. In questo esempio, «documento» è il nome del documento a cui si applicano i metadati e «estensione» è l'estensione del file del documento. L'ID del documento deve essere univoco in<document>.<extension>.metadata.json.

Il contenuto del file JSON segue questo modello. Tutti gli attributi sono facoltativi. Se non si specifica il valore_source_uri, i link restituiti daiAmazon Kendra risultati della ricerca rimandano alAmazon S3 bucket che contiene il documento.

{
    "DocumentId": "document ID",
    "Attributes": {
        "_category": "document category",
        "_created_at": "ISO 8601 encoded string",
        "_last_updated_at": "ISO 8601 encoded string",
        "_source_uri": "document URI",
        "_version": "file version",
        "_view_count": number of times document has been viewed,
        "custom attribute key": "custom attribute value",
        additional custom attributes
    },
    "AccessControlList": [
         {
             "Name": "user name",
             "Type": "GROUP | USER",
             "Access": "ALLOW | DENY"
         }
    ],
    "Title": "document title",
    "ContentType": "For example HTML | PDF. For supported content types, see Types of documents."
}

I campi_created_at e i_last_updated_at metadati sono date codificate ISO 8601. Ad esempio, 2012-03-25T 12:30:10 + 01:00 è il formato data/ora ISO 8601 per il 25 marzo 2012 alle 12:30 (più 10 secondi) nel fuso orario dell'Europa centrale (CET).

È possibile aggiungere ulteriori informazioni alAttributes campo relative a un documento che si utilizza per filtrare le interrogazioni o per raggruppare le risposte alle interrogazioni. Per ulteriori informazioni, consulta Creazione di documenti personalizzati.

È possibile utilizzare ilAccessControlList campo per filtrare la risposta da una query. In questo modo, solo determinati utenti e gruppi hanno accesso ai documenti. Per ulteriori informazioni, consulta Filtraggio in base al contesto dell'utente.
