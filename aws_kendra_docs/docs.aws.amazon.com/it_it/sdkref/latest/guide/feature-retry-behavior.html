Nuovo tentativo

Il comportamento di ripetizione include le impostazioni relative al modo in cui gli SDK tentano di ripristinare gli errori derivanti dalle richieste inviate aServizi AWS.

Configura questa funzionalità utilizzando quanto segue:

max_attempts- impostazione deiAWSconfig file condivisi
AWS_MAX_ATTEMPTS- variabile di ambiente

Specifica il numero massimo di tentativi da effettuare su una richiesta.

Valore predefinito: se questo valore non è specificato, il suo valore predefinito dipende dal valore dell'retry_modeimpostazione:

  • In casoretry_modelegacy affermativo: utilizza un valore predefinito specifico per il tuo SDK (consulta la guida SDK specifica o la base di codice del tuo SDK permax_attempts impostazione predefinita).

  • In casoretry_modestandard affermativo, effettua tre tentativi.

  • In casoretry_modeadaptive affermativo, effettua tre tentativi.

Valori validi: numero maggiore di 0.

retry_mode- impostazione deiAWSconfig file condivisi
AWS_RETRY_MODE- variabile di ambiente

Specifica in che modo l'SDK o lo strumento di sviluppo tenta di riprovare.

Valore predefinito:legacy è la strategia di ripetizione predefinita.

Valori validi:

  • legacy— Specifico per il tuo SDK (consulta la guida SDK specifica o la base di codice del tuo SDK).

  • standard— Il set standard di regole per i tentativi ripetuti negliAWS SDK. Questa modalità include un set standard di errori che vengono ritentati e il supporto per le quote di ripetizione. Il numero massimo predefinito di tentativi con questa modalità è tre, a meno che nonmax_attempts sia configurato in modo esplicito.

  • adaptive— Una modalità sperimentale di nuovo tentativo che include la funzionalità della modalità standard ma include la limitazione automatica sul lato client. Poiché questa modalità è sperimentale, potrebbe cambiare comportamento in future.

Di seguito è riportato lo pseudocodice di alto livello per entrambe le modalitàstandard eadaptive retry:

MakeSDKRequest() {
  attempts = 0
  loop {
    GetSendToken()
    response = SendHTTPRequest()
    RequestBookkeeping(response)
    if not Retryable(response)
      return response
    attempts += 1
    if attempts >= MAX_ATTEMPTS:
      return response
    if not HasRetryQuota(response)
      return response
    delay = ExponentialBackoff(attempts)
    sleep(delay)
  }
}

Di seguito sono riportati ulteriori dettagli sui componenti utilizzati nello pseudocodice:

GetSendToken:

I bucket di token vengono utilizzati solo in modalitàadaptive Riprova. I bucket di token impongono una frequenza massima di richieste richiedendo che un token sia disponibile per avviare una richiesta. Il client SDK è configurabile per fallire rapidamente la richiesta o bloccarlo finché non diventa disponibile un token.

Client Side Rate Limiting è un algoritmo che inizialmente consente di effettuare richieste in qualsiasi momento fino al limite del token. Tuttavia, dopo che viene rilevata una risposta limitata, il client rate-of-request viene limitato di conseguenza. Anche l'indennità simbolica viene aumentata di conseguenza se vengono ricevute risposte positive.

Grazie alla limitazione adattiva della velocità, gli SDK possono rallentare la velocità di invio delle richieste per adattarsi meglio alla capacità diServizi AWS.

SendHTTPRequest:

La maggior parte degliAWS SDK utilizza una libreria HTTP che utilizza pool di connessioni in modo da poter riutilizzare una connessione esistente quando si effettua una richiesta HTTP. In genere, le connessioni vengono riutilizzate quando si riprovano le richieste a causa di errori di limitazione. Le richieste non vengono riutilizzate quando si riprova a causa di errori transitori.

RequestBookkeeping:

La quota di nuovi tentativi deve essere aggiornata se la richiesta ha esito positivo. Solo per la modalità diadaptive riprova, la variabile di statomaxsendrate viene aggiornata in base al tipo di risposta ricevuta.

Retryable:

Questo passaggio determina se una risposta può essere ritentata in base a quanto segue:

  • Codice di stato HTTP .

  • Il codice di errore restituito dal servizio.

  • Errori di connessione, definiti come qualsiasi errore ricevuto dall'SDK in cui non viene ricevuta una risposta HTTP dal servizio.

Gli errori transitori (codici di stato HTTP 400, 408, 500, 502, 503 e 504) e gli errori di limitazione (codici di stato HTTP 400, 403, 429, 502, 503 e 509) possono essere tutti potenzialmente ritentati. Il comportamento dei tentativi dell'SDK viene determinato in combinazione con codici di errore o altri dati del servizio.

MAX_ATTEMPTS:

Specificato dall'impostazioneconfig del file o dalla variabile di ambiente.

HasRetryQuota

Questo passaggio limita le richieste di nuovi tentativi richiedendo che un token sia disponibile nel bucket delle quote per i tentativi successivi. I periodi fissi di quote riprovate sono un meccanismo per prevenire nuovi tentativi che difficilmente avranno esito positivo. Queste quote dipendono dall'SDK, spesso dipendono dal client e talvolta dipendono anche dagli endpoint del servizio. I token di quota per i tentativi disponibili vengono rimossi quando le richieste falliscono per vari motivi e ripristinati quando hanno esito positivo. Quando non rimane alcun token, il ciclo di tentativi viene chiuso.

ExponentialBackoff

Per un errore che può essere ripetuto, il ritardo nei tentativi viene calcolato utilizzando un backoff esponenziale troncato. Gli SDK utilizzano un backoff esponenziale binario troncato con jitter. Il seguente algoritmo mostra come viene definita la durata del sonno, in secondi, per una risposta alla richiestai: 

seconds_to_sleep_i = min(b*r^i, MAX_BACKOFF)

Nell'algoritmo precedente, si applicano i seguenti valori:

b = random number within the range of: 0 <= b <= 1

r = 2

MAX_BACKOFF = 20 secondsper la maggior parte degli SDK. Consulta la guida SDK o il codice sorgente specifico per la conferma.
