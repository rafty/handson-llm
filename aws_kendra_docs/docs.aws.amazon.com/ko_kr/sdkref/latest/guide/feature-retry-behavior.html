재시도 동작

재시도 동작에는 SDK가 요청으로 인한 장애로부터 복구를 시도하는 방법과 관련된 설정이 포함됩니다AWS 서비스.

다음을 사용하여 이 기능을 구성하십시오.

max_attempts- 공유AWSconfig 파일 설정
AWS_MAX_ATTEMPTS- 환경 변수

요청에서 시도할 수 있는 최대 횟수를 지정합니다.

기본값: 이 값을 지정하지 않은 경우 기본값은retry_mode 설정값에 따라 달라집니다.

  • retry_modeIf islegacy — SDK 고유의 기본값을 사용합니다 (max_attempts기본값은 특정 SDK 가이드 또는 SDK 코드베이스를 확인하세요).

  • 있는retry_mode 경우standard — 세 번 시도합니다.

  • 있는retry_mode 경우adaptive — 세 번 시도합니다.

유효한 값: 0보다 큰 숫자.

retry_mode- 공유AWSconfig 파일 설정
AWS_RETRY_MODE- 환경 변수

SDK 또는 개발자 도구가 재시도를 시도하는 방법을 지정합니다.

기본값:legacy 은 기본 재시도 전략입니다.

유효한 값:

  • legacy— SDK에만 해당됩니다 (특정 SDK 가이드 또는 SDK의 코드베이스를 확인하세요).

  • standard—AWS SDK 전반의 표준 재시도 규칙 세트입니다. 이 모드에는 재시도되는 표준 오류 집합과 재시도 할당량 지원이 포함됩니다. 명시적으로 구성하지 않는 한 이 모드의 기본 최대 시도 횟수는max_attempts 3회입니다.

  • adaptive— 표준 모드의 기능을 포함하지만 자동 클라이언트측 스로틀링을 포함하는 실험용 재시도 모드입니다. 이 모드는 실험용이므로 future 동작이 변경될 수 있습니다.

다음은 재시도standard 모드와adaptive 재시도 모드 모두에 대한 상위 수준 유사 코드입니다.

MakeSDKRequest() {
  attempts = 0
  loop {
    GetSendToken()
    response = SendHTTPRequest()
    RequestBookkeeping(response)
    if not Retryable(response)
      return response
    attempts += 1
    if attempts >= MAX_ATTEMPTS:
      return response
    if not HasRetryQuota(response)
      return response
    delay = ExponentialBackoff(attempts)
    sleep(delay)
  }
}

다음은 유사 코드에 사용되는 구성 요소에 대한 자세한 내용입니다.

GetSendToken:

토큰 버킷은adaptive 재시도 모드에서만 사용됩니다. 토큰 버킷은 요청을 시작하기 위해 토큰을 사용할 수 있도록 요구하여 최대 요청 비율을 적용합니다. SDK 클라이언트는 요청을 빠르게 실패시키거나 토큰을 사용할 수 있을 때까지 차단하도록 구성할 수 있습니다.

클라이언트 측 속도 제한은 처음에 토큰 허용 한도 내에서 원하는 요율로 요청이 이루어질 수 있도록 하는 알고리즘입니다. 그러나 스로틀링된 응답이 감지되면 rate-of-request 클라이언트가 이에 따라 제한됩니다. 성공적인 응답을 받으면 그에 따라 토큰 허용량도 증가합니다.

적응형 속도 제한을 사용하면 SDK는 요청 전송 속도를 늦추어 용량을 더 잘 수용할 수AWS 서비스 있습니다.

SendHTTPRequest:

대부분의AWS SDK는 연결 풀을 사용하는 HTTP 라이브러리를 사용하므로 HTTP 요청 시 기존 연결을 재사용할 수 있습니다. 일반적으로 전송률 조절 오류로 인해 요청을 재시도할 때 연결이 재사용됩니다. 일시적인 오류로 인해 재시도할 때 요청이 재사용되지 않습니다.

RequestBookkeeping:

요청이 성공하면 재시도 할당량을 업데이트해야 합니다. adaptive재시도 모드의 경우에만 수신된 응답 유형에 따라 상태 변수가maxsendrate 업데이트됩니다.

Retryable:

이 단계는 다음을 기반으로 응답을 재시도할 수 있는지 여부를 결정합니다.

  • HTTP 상태 코드 .

  • 서비스에서 반환된 오류 코드입니다.

  • 연결 오류는 서비스로부터 HTTP 응답을 받지 못한 SDK에 수신된 오류로 정의됩니다.

일시적인 오류 (HTTP 상태 코드 400, 408, 500, 502, 503 및 504) 및 스로틀 오류 (HTTP 상태 코드 400, 403, 429, 502, 503 및 509) 는 모두 재시도될 수 있습니다. SDK 재시도 동작은 오류 코드 또는 서비스의 기타 데이터와 함께 결정됩니다.

MAX_ATTEMPTS:

config파일 설정 또는 환경 변수로 지정됩니다.

HasRetryQuota

이 단계에서는 재시도 할당량 버킷에서 토큰을 사용할 수 있도록 요구하여 재시도 요청을 조절합니다. 재시도 할당량 버킷은 성공 가능성이 낮은 재시도를 방지하는 메커니즘입니다. 이러한 할당량은 SDK에 따라 다르며 클라이언트에 따라 달라지는 경우가 많으며 때로는 서비스 엔드포인트에 의존하기도 합니다. 사용 가능한 재시도 할당량 토큰은 여러 가지 이유로 요청이 실패하면 제거되고 성공하면 보충됩니다. 토큰이 남아 있지 않으면 재시도 루프가 종료됩니다.

ExponentialBackoff

재시도할 수 있는 오류의 경우 잘린 지수 백오프를 사용하여 재시도 지연을 계산합니다. SDK는 지터가 포함된 잘린 이진 지수 백오프를 사용합니다. 다음 알고리즘은 요청에 대한 응답에 대해 잠자기 시간을 초 단위로 정의하는 방법을 보여줍니다i.

seconds_to_sleep_i = min(b*r^i, MAX_BACKOFF)

위 알고리즘에서는 다음 값이 적용됩니다.

b = random number within the range of: 0 <= b <= 1

r = 2

MAX_BACKOFF = 20 seconds대부분의 SDK에서 사용할 수 있습니다. 확인을 위해 특정 SDK 가이드 또는 소스 코드를 참조하십시오.
