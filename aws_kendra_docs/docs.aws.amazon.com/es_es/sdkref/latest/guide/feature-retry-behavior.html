Comportamiento de reintentar

El comportamiento de reintento incluye la configuración relativa a la forma en que los SDK intentan recuperarse de los errores derivados de las solicitudes realizadas aServicios de AWS.

Configure esta funcionalidad mediante lo siguiente:

max_attempts- configuración deAWSconfig archivos compartidos
AWS_MAX_ATTEMPTS- variable de entorno

Especifica el número máximo de intentos que se deben realizar en una solicitud.

Valor predeterminado: si no se especifica este valor, su valor predeterminado depende del valor de laretry_mode configuración:

  • Siretry_mode es asílegacy, usa un valor predeterminado específico de tu SDK (consulta la guía específica del SDK o la base de códigos del SDK para conocer los valoresmax_attempts predeterminados).

  • Siretry_mode esstandard así, hace tres intentos.

  • Siretry_mode esadaptive así, hace tres intentos.

Valores válidos: número mayor que 0.

retry_mode- configuración deAWSconfig archivos compartidos
AWS_RETRY_MODE- variable de entorno

Especifica la forma en que el SDK o la herramienta de desarrollador intentan volver a intentarlo.

Valor predeterminado:legacy es la estrategia de reintento predeterminada.

Valores válidos:

  • legacy— Específico para tu SDK (consulta la guía específica del SDK o la base de código de tu SDK).

  • standard— El conjunto estándar de reglas de reintento en todosAWS los SDK. Este modo incluye un conjunto estándar de errores que se reintentan y admite cuotas de reintentos. El número máximo predeterminado de intentos con este modo es de tres, a menos quemax_attempts esté configurado explícitamente.

  • adaptive— Un modo de reintento experimental que incluye la funcionalidad del modo estándar, pero incluye la limitación automática del lado del cliente. Dado que este modo es experimental, podría cambiar su comportamiento en el future.

A continuación se muestra el pseudocódigo de alto nivel para los modosstandard yadaptive reintento:

MakeSDKRequest() {
  attempts = 0
  loop {
    GetSendToken()
    response = SendHTTPRequest()
    RequestBookkeeping(response)
    if not Retryable(response)
      return response
    attempts += 1
    if attempts >= MAX_ATTEMPTS:
      return response
    if not HasRetryQuota(response)
      return response
    delay = ExponentialBackoff(attempts)
    sleep(delay)
  }
}

A continuación se muestran más detalles sobre los componentes que se utilizan en el pseudocódigo:

GetSendToken:

Los depósitos de fichas solo se utilizan en el modo deadaptive reintento. Los depósitos de tokens imponen una tasa máxima de solicitudes al requerir que un token esté disponible para iniciar una solicitud. El cliente del SDK se puede configurar para anular rápidamente la solicitud o bloquearla hasta que haya un token disponible.

La limitación de velocidad del lado del cliente es un algoritmo que inicialmente permite realizar solicitudes a cualquier velocidad hasta el límite permitido por el token. Sin embargo, una vez que se detecta una respuesta restringida, el cliente rate-of-request se limita en consecuencia. La asignación de fichas también se incrementa en consecuencia si se reciben respuestas satisfactorias.

Con la limitación de velocidad adaptativa, los SDK pueden reducir la velocidad a la que se envían las solicitudes para adaptarse mejor a la capacidad deServicios de AWS.

SendHTTPRequest:

La mayoría deAWS los SDK utilizan una biblioteca HTTP que usa grupos de conexiones para que puedas reutilizar una conexión existente al realizar una solicitud HTTP. Por lo general, las conexiones se reutilizan al reintentar las solicitudes debido a errores de limitación. Las solicitudes no se reutilizan al volver a intentarlo debido a errores transitorios.

RequestBookkeeping:

La cuota de reintentos debe actualizarse si la solicitud se realiza correctamente. Solo para el modo deadaptive reintento, la variable de estadomaxsendrate se actualiza en función del tipo de respuesta recibida.

Retryable:

Este paso determina si se puede volver a intentar una respuesta en función de lo siguiente:

  • El código de estado HTTP .

  • El código de error devuelto por el servicio.

  • Errores de conexión, definidos como cualquier error recibido por el SDK en el que no se reciba una respuesta HTTP del servicio.

Es posible que se vuelvan a intentar los errores transitorios (códigos de estado HTTP 400, 408, 500, 502, 503 y 504) y los errores de limitación (códigos de estado HTTP 400, 403, 429, 502, 503 y 509). El comportamiento de reintento del SDK se determina en combinación con códigos de error u otros datos del servicio.

MAX_ATTEMPTS:

Especificado por la configuración delconfig archivo o la variable de entorno.

HasRetryQuota

Este paso reduce las solicitudes de reintentos al requerir que haya un token disponible en el depósito de cuotas de reintentos. Los cubos de cuotas de reintentos son un mecanismo para evitar reintentos que probablemente no tengan éxito. Estas cuotas dependen del SDK, suelen depender del cliente y, a veces, incluso de los puntos finales del servicio. Los tokens de cuota de reintentos disponibles se eliminan cuando las solicitudes fallan por diversos motivos y se reponen cuando tienen éxito. Cuando no quede ningún identificador, se cierra el ciclo de reintentos.

ExponentialBackoff

En el caso de un error que se pueda volver a intentar, el retraso en el reintento se calcula mediante un retraso exponencial truncado. Los SDK utilizan un retroceso exponencial binario truncado con fluctuación. El siguiente algoritmo muestra cómo se define la cantidad de tiempo de sueño, en segundos, para una respuesta a una solicitudi: 

seconds_to_sleep_i = min(b*r^i, MAX_BACKOFF)

En el algoritmo anterior, se aplican los siguientes valores:

b = random number within the range of: 0 <= b <= 1

r = 2

MAX_BACKOFF = 20 secondspara la mayoría de los SDK. Consulte la guía específica del SDK o el código fuente para confirmarlo.
