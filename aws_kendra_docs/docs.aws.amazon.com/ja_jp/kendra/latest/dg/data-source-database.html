Amazon RDS/Aurora

データベースデータソースを使用して、データベースに保存されているドキュメントにインデックスを作成することができます。データベースの接続情報を入力すると、Amazon Kendraドキュメントを接続しドキュメントにインデックスを作成します。

Amazon Kendraでは、以下のデータベースがサポートされています。

  • Amazon Aurora MySQL

  • Amazon Aurora PostgreSQL

  • Amazon RDS for MySQL

  • Amazon RDS for PostgreSQL

注記

サーバーレス Aurora データベースはサポートされていません。

Amazon Kendraコンソールと DatabaseConfigurationAPIAmazon Kendra を使用してデータベースデータソースに接続できます。

サポートされている機能

Amazon Kendraデータベースデータソースコネクタは以下の機能をサポートしています。

  • フィールドマッピング

  • ユーザーコンテキストフィルタリング

  • Virtual Private Cloud (VPC)

前提条件

Amazon Kendraを使用してデータベースデータソースのインデックスを作成する前に、AWSデータベースとアカウントでこれらの変更を行います。

データベースに、次のものがあることを確認してください。

  • データソースの基本認証情報として、ユーザーとパスワードのペアを記録しました。

  • ホスト名、ポート番号、ホストアドレス、データベースデータが含まれているデータテーブル内のデータテーブルの名前をコピーしました。PostgreSQL の場合、データテーブルはパブリックテーブルでなければなりません。

    注記

    ホストとポートは、Amazon Kendraインターネット上のデータベースサーバーの場所を指示します。データベース名とテーブル名によって、Amazon Kendraデータベースサーバー上のドキュメントデータの場所が指定されます。

  • ドキュメントデータ、ドキュメント ID、カスタムインデックスフィールドにマッピングされるオプションのデータテーブル内の列の名前、ドキュメント ID、カスタムインデックスフィールドにマッピングされるオプションのデータテーブル列にマッピングされるオプションのデータテーブル内の列の名前、ドキュメント ID、カスタムインデックスフィールドにマッピングされるオプションのデータテーブル列の列名。Amazon Kendra予約済みフィールド名をテーブルの列にマッピングできます。

  • MySQLAmazon RDS と別のタイプのどちらを使用するかなど、データベースエンジンのタイプ情報をコピーしました。

  • 各ドキュメントがデータベース内および同じインデックスに使用する予定の他のデータソース間で一意であることを確認しました。インデックスに使用する各データソースには、データソース間で同じドキュメントが含まれていてはなりません。ドキュメント ID はインデックス全体に適用され、インデックスごとに一意でなければなりません。

AWSアカウントには、次のものがあることを確認してください。

  • Amazon Kendraインデックスを作成し、API を使用している場合は、インデックス ID を記録しました。

  • IAMデータソースのロールを作成し、API を使用している場合は、IAMロールの ARN を記録しました。

  • AWSSecrets Managerデータベース認証情報をシークレットに保存し、API を使用している場合は、シークレットの ARN を記録しました。

    注記

    認証情報とシークレットは定期的に更新またはローテーションしてください。セキュリティに必要なアクセスレベルのみを指定してください。データソース、およびコネクタバージョン 1.0 と 2.0 (該当する場合) 間で認証情報とシークレットを再利用することはお勧めしません。

IAM既存のロールまたはシークレットがない場合は、データベースデータソースをに接続するときに、IAMSecrets Managerコンソールを使用して新しいロールとシークレットを作成できますAmazon Kendra。API を使用している場合は、IAMSecrets Manager既存のロールとシークレットの ARN、およびインデックス ID を指定する必要があります。

接続手順

Amazon Kendraデータベースデータソースに接続するには、Amazon Kendraデータにアクセスできるように、データベースデータソースの必要な詳細を入力する必要があります。データベースをまだ設定していない場合は、Amazon Kendraを参照してください前提条件。

Console

Amazon Kendraデータベースに接続するには

  1. AWS マネジメントコンソールにサインインし、Amazon Kendra コンソールを開きます。

  2. 左のナビゲーションペインで、[インデックス] を選択し、インデックスの一覧から使用するインデックスを選択します。

    注記

    ユーザーアクセス制御設定は、インデックス設定で設定または編集できます。

  3. 「はじめに」ページで、「データソースを追加」を選択します。

  4. [データソースの追加] ページで、[データベースコネクタ] を選択し、[データソースの追加] を選択します。

  5. [データソースの詳細を指定] ページで次の情報を入力します。

    1. [名前と説明] の [データソース名] に、データソースの名前を入力します。ハイフンは使用できますが、スペースは含められません。

    2. (オプション) [説明]-データソースの説明を入力します。

    3. 「言語」(「デフォルト言語」)-索引用にドキュメントをフィルタリングする言語。特に指定しない限り、言語はデフォルトで英語に設定されます。メタデータで指定された言語は、選択した言語よりも優先されます。

    4. [タグ] で、[Add new tag] で、リソースを検索してフィルタリングしたり、AWSコストを追跡したりできます。

    5. [Next] (次へ) を選択します。

  6. 「アクセスとセキュリティの定義」ページで、次の情報を入力します。

    1. エンドポイント — DNS ホスト名、IPv4 アドレス、IPv4 アドレス、IPv6 アドレス。

    2. ポート：ポート番号。

    3. データベース — データベース名。

    4. テーブル名 — テーブル名。

    5. [認証タイプ] で、[既存] と [新規] のどちらかを選択して、データベース認証情報を保存します。新しいシークレットを作成することを選択すると、AWSSecrets Managerシークレットウィンドウが開きます。

      1. 「AWSSecrets Managerシークレットの作成」ウィンドウに次の情報を入力します。

        1. シークレット名 — シークレットの名前。プレフィックス「AmazonKendra-database-」は、シークレット名に自動的に追加されます。

        2. [ユーザー名] と [パスワード] の場合-データベースアカウントの認証資格値を入力します。

        3. [認証を保存] を選択します。

    6. [VVirtual Private Cloud (VPC) — VPC を選択できます。その場合は、サブネットと VPC セキュリティグループを追加する必要があります。

      注記

      プライベートサブネットを使用する必要があります。RDS インスタンスが VPC のパブリックサブネット内にある場合は、パブリックサブネット内の NAT ゲートウェイへのアウトバウンドアクセス権を持つプライベートサブネットを作成できます。VPC 構成で提供されるサブネットは、米国西部 (オレゴン)、米国東部 (北バージニア)、欧州 (アイルランド) のどちらかに存在する必要があります。

    7. IAM ロール — 既存の IAM ロールを選択して割り当てるか、新しい IAM ロールを作成します。

      注記

      インデックスに使用される IAM ロールはデータソースには使用できません。既存のロールがインデックスや FAQ に使用されているかどうかわからない場合は、エラーを避けるため [新しいロールを作成する] を選択してください。

    8. [Next] (次へ) を選択します。

  7. [Sync 設定の設定] ページで次の情報を入力します。

    1. ユースケースに基づいて、Aurora MySQL 、MySQL、Aurora PostgreSQL 、PostgreSQL の中から選択してください。

    2. SQL 識別子を二重引用符で囲む-選択すると、SQL 識別子を二重引用符で囲みます。たとえば、「columnName」。

    3. ACL 列と変更検出列-Amazon Kendra 変更検出リストとアクセス制御リストに使用する列の設定に使用します。

    4. [同期実行スケジュール] の [頻度] で、Amazon Kendraデータソースと同期する頻度を選択します。

    5. [Next] (次へ) を選択します。

  8. [フィールドマッピングの設定] ページで次の情報を入力します。

    1. Amazon Kendraデフォルトフィールドマッピング —Amazon Kendra 生成されたデフォルトデータソースフィールドから、インデックスにマッピングするフィールドを選択します。document_idとには、データベース列の値を追加する必要があります。document_body

    2. カスタムフィールドマッピング — カスタムデータソースフィールドを追加して、マッピング先のインデックスフィールド名とフィールドデータタイプを作成します。

    3. [Next] (次へ) を選択します。

  9. [確認と作成] ページで、入力した情報が正しいことを確認し、[データソースの追加] を選択します。また、このページで情報を編集することもできます。データソースが正常に追加されると、データソースページに表示されます。

API

Amazon Kendraデータベースに接続するには

次の DatabaseConfigurationAPI を指定する必要があります。

  • ColumnConfiguration—インデックスがデータベースからドキュメント情報を取得する必要がある場所に関する情報。詳細については、「ColumnConfiguration」を参照してください。DocumentDataColumnNameDocumentIdColumnNameおよびフィールドを指定する必要があります。DocumentIdColumnName フィールドにマッピングされた列は整数の列である必要があります。次の例は、データベースデータソースの単純な列構成を示しています。

    "ColumnConfiguration": {
    "ChangeDetectingColumns": [
        "LastUpdateDate",
        "LastUpdateTime"
    ],
    "DocumentDataColumnName": "TextColumn",
    "DocumentIdColumnName": "IdentifierColumn",
    "DocoumentTitleColumnName": "TitleColumn",
    "FieldMappings": [
        {
            "DataSourceFieldName": "AbstractColumn",
            "IndexFieldName": "Abstract"
        }
    ]
}

  • ConnectionConfiguration—データベースを実行するデータベースエンジンのタイプ。DatabaseHostフィールドはデータベースのAmazon Relational Database Service (Amazon RDS) インスタンスエンドポイントでなければなりません。クラスターエンドポイントを使用しないでください。

  • DatabaseEngineType—データベースに接続するために必要な構成情報。詳細については、「ConnectionConfiguration」を参照してください。

  • シークレットの Amazon リソースネーム (ARN) —Secrets Manager データベースアカウントの認証情報を含むシークレットの Amazon リソースネーム (ARN) を指定します。シークレットは、次のキーを持つ JSON 構造で格納されます。

    {
    "username": "user name",
    "password": "password"
}

    シークレットにはより多くの情報を含めることができます。次の図にデータベース設定の例を示します。

    
"DatabaseConfiguration": {
"ConnectionConfiguration": {
"DatabaseHost": "host.subdomain.domain.tld",
        "DatabaseName": "DocumentDatabase",
        "DatabasePort": 3306,
        "SecretArn": "arn:aws:secretmanager:region:account ID:secret/secret name",
        "TableName": "DocumentTable"
    }
}
    注記

    認証情報とシークレットは定期的に更新またはローテーションしてください。セキュリティに必要なアクセスレベルのみを指定してください。データソース、およびコネクタバージョン 1.0 と 2.0 (該当する場合) 間で認証情報とシークレットを再利用することはお勧めしません。

  • IAM ロール —RoleArnCreateDataSource 呼び出すタイミングを指定して、IAMSecrets Managerシークレットにアクセスする権限をロールに与え、データベースコネクタに必要なパブリック APIAmazon Kendra を呼び出すことができます。詳細については、「データベースデータソースの IAM ロール」を参照してください。

オプションで、次のオプション機能を追加することもできます。

  • Virtual Private Cloud (VPC) —VpcConfiguration データソース設定の一部として指定します。「VPCAmazon Kendra を使用するように設定する」を参照してください。

    注記

    プライベートサブネットのみを使用する必要があります。RDS インスタンスが VPC のパブリックサブネット内にある場合は、パブリックサブネット内の NAT ゲートウェイへのアウトバウンドアクセス権を持つプライベートサブネットを作成できます。VPC 構成で提供されるサブネットは、米国西部 (オレゴン)、米国東部 (北バージニア)、欧州 (アイルランド) のどちらかに存在する必要があります。

  • フィールドマッピング —Amazon Kendra データベースデータソースフィールドをインデックスフィールドにマッピングすることを選択します。詳細については、データソースフィールドのマッピングを参照してください。

  • ユーザーコンテキストフィルタリング — デフォルトで、データソースのアクセス制御リスト (ACL)Amazon Kendra をクロールします。ACL 情報は、ドキュメントへのユーザーまたはグループのアクセス権に基づいて検索結果をフィルタリングします。詳細については、データベースデータソースのユーザーコンテキストフィルタリングを参照してください。
