Slack

Slackは、ユーザーがさまざまなパブリックチャネルやプライベートチャネルを通じてメッセージや添付ファイルを送信できるエンタープライズコミュニケーションアプリです。Slackのパブリックチャンネルとプライベートチャンネル、ボットとアーカイブメッセージ、ファイルと添付ファイル、ダイレクトメッセージとグループメッセージのインデックスを作成できます。Amazon Kendraフィルタリングする特定のコンテンツを選択することもできます。

Amazon Kendraコンソールと SlackConfigurationAPI を使用して SlackAmazon Kendra データソースに接続できます。

サポートされている機能

Amazon KendraSlack データソースコネクタは以下の機能をサポートしています。

  • 変更ログ

  • フィールドマッピング

  • ユーザーコンテキストのフィルター

  • 包含/除外フィルター

  • Virtual Private Cloud (VPC)

前提条件

Amazon Kendraを使用して Slack データソースのインデックスを作成する前に、SlackAWS とアカウントでこれらの変更を行います。

Slack では、以下のものが揃っていることを確認してください。

  • Slack ボットユーザー OAuth トークンまたは Slack ユーザー OAuth トークンを作成しました。どちらのトークンでも SlackAmazon Kendra データソースに接続できます。詳細については、アクセストークンに関するSlackのドキュメントを参照してください。

    注記

    ボットトークンを Slack の認証情報の一部として使用する場合、ダイレクトメッセージやグループメッセージをインデックスに登録することはできません。インデックスに登録するチャンネルにボットトークンを追加する必要があります。

  • Slack ワークスペースのメインページ URL にある Slack ワークスペースのチーム ID を書き留めました。たとえば、https://app.slack.com/client/T0123456789/... ここで、T0123456789 はチーム ID です。

  • 以下の [OAuth スコープ/読み取り] 権限を追加しました。

    • チャネル:履歴

    • チャネル:読み込み

    • グループ:履歴

    • グループ:読み込み

    • IM: 歴史

    • MPIM: 履歴

    • MPIM: 読み込み

    • team:read

    • ユーザプロファイル:読み込み

    • ユーザ:読み取り

    • 絵文字:読んだ

    • ファイル:読み込み

    • ユーザグループ:読み込み

  • 各ドキュメントが Slack 内および同じインデックスに使用する予定の他のデータソースとは一意であることを確認しました。インデックスに使用する各データソースには、データソース間で同じドキュメントが含まれていてはなりません。ドキュメント ID はインデックス全体に適用され、インデックスごとに一意でなければなりません。

AWSアカウントには、次のものがあることを確認してください。

  • Amazon Kendraインデックスを作成し、API を使用している場合は、インデックス ID を記録しました。

  • IAMデータソースのロールを作成し、API を使用している場合は、IAMロールの ARN を記録しました。

  • SlackAWSSecrets Manager の認証情報をシークレットに保存し、API を使用している場合は、シークレットの ARN を記録しました。

    注記

    認証情報とシークレットは定期的に更新またはローテーションしてください。セキュリティに必要なアクセスレベルのみを指定してください。データソース、およびコネクタバージョン 1.0 と 2.0 (該当する場合) 間で認証情報とシークレットを再利用することはお勧めしません。

IAM既存のロールやシークレットがない場合は、Slack データソースをに接続するときに、IAMSecrets Managerコンソールを使用して新しいロールとシークレットを作成できますAmazon Kendra。API を使用している場合は、IAMSecrets Manager既存のロールとシークレットの ARN とインデックス ID を指定する必要があります。

接続手順

SlackAmazon Kendra データソースに接続するには、Amazon Kendraデータにアクセスできるように Slack データソースの必要な詳細を入力する必要があります。Slack をまだ設定していない場合は、Amazon Kendraを参照してください前提条件。

Console

SlackAmazon Kendra に接続するには

  1. AWS マネジメントコンソールにサインインし、Amazon Kendra コンソールを開きます。

  2. 左側のナビゲーションペインで、[インデックス] を選択し、インデックスのリストから使用するインデックスを選択します。

    注記

    ユーザーアクセス制御設定は、インデックス設定で設定または編集できます。

  3. 「はじめに」ページで、「データソースを追加」を選択します。

  4. [データソースの追加] ページで [Slack コネクタ] を選択し、[データソースの追加] を選択します。

  5. [データソースの詳細を指定] ページで、次の情報を入力します。

    1. [名前と説明] の [データソース名] に、データソースの名前を入力します。ハイフンは使用できますが、スペースは使用できません。

    2. (オプション) 説明-オプションで、データソースの説明を入力します。

    3. 「言語」(「デフォルト言語」)-索引用にドキュメントをフィルタリングする言語。特に指定しない限り、言語はデフォルトで英語に設定されます。メタデータで指定された言語は、選択した言語よりも優先されます。

    4. [タグ] で、[Add new tag] で、リソースを検索してフィルタリングしたり、AWSコストを追跡したりできます。

    5. [Next] (次へ) を選択します。

  6. 「アクセスとセキュリティの定義」ページで、次の情報を入力します。

    1. Slack ワークスペースのチーム ID — Slack ワークスペースのチーム ID。

    2. AWSSecrets Managersecret —既存のシークレットを選択するか、Secrets Manager新しいシークレットを作成して Slack 認証情報を保存します。新しいシークレットを作成することを選択すると、AWSSecrets Managerシークレットウィンドウが開きます。

      1. 「AWSSecrets Managerシークレットの作成」ウィンドウに次の情報を入力します。

        1. シークレット名 — シークレットの名前。シークレット名には「AmazonKendra-Slack-」というプレフィックスが自動的に追加されます。

        2. Slack トークンの場合 — Slack アカウントで作成した認証情報の値を入力します。

      2. [Save] (保存) を選択します。

    3. [VVirtual Private Cloud (VPC) — VPC を選択することができます。その場合は、サブネットと VPC セキュリティグループを追加する必要があります。

    4. IAM ロール — 既存の IAM ロールを選択するか、新しい IAM ロールを作成して、リポジトリの認証情報にアクセスしてコンテンツをインデックス作成します。

      注記

      インデックスに使用される IAM ロールはデータソースには使用できません。既存のロールがインデックスや FAQ に使用されているかどうかわからない場合は、エラーを避けるため [新しいロールを作成する] を選択してください。

    5. [Next] (次へ) を選択します。

  7. [同期設定の設定] ページで、次の情報を入力します。

    1. クロールするコンテンツのタイプを選択 — クロールする Slack エンティティまたはコンテンツタイプ。

    2. 変更ログ — 選択すると、すべてのファイルを同期する代わりにインデックスを更新できます。

    3. 正規表現パターン — 特定のファイルを包含または除外する正規表現パターン。最大 100 個のパターンを追加できます。

    4. クロール開始日の選択-Amazon Kendra データのクロールを開始する日付を指定します。

    5. 同期実行スケジュール (頻度)-Amazon Kendra データソースと同期する頻度

    6. [Next] (次へ) を選択します。

  8. [フィールドマッピングの設定] ページで、次の情報を入力します。

    1. Slack フィールドマッピングの場合 —Amazon Kendra 生成されたデフォルトデータソースフィールドから、インデックスにマッピングするフィールドを選択します。

    2. フィールドを追加 — カスタムデータソースフィールドを追加して、マッピング先のインデックスフィールド名とフィールドデータタイプを作成します。

    3. [Next] (次へ) を選択します。

  9. [確認と作成] ページで、入力した情報が正しいことを確認し、[データソースの追加] を選択します。また、このページで情報を編集することもできます。データソースが正常に追加されると、データソースページに表示されます。

API

SlackAmazon Kendra に接続するには

SlackConfigurationAPI を使用して以下を指定する必要があります。

  • Slack ワークスペースのチーム ID — Slack のメインページ URL からコピーした Slack チーム ID。

  • インデックスを作成するエンティティのリスト — 公開チャンネルとプライベートチャンネル、グループメッセージ、ダイレクトメッセージをインデックスに登録するかどうかAmazon Kendra。

  • クロール日 — Slack ワークスペースチームからのデータのクロールを開始する日付。日付はこの形式に従う必要があります: yyyy-mm-dd.

  • シークレットの Amazon リソースネーム (ARN) — SlackSecrets Manager アカウントの認証情報を含むシークレットの Amazon リソースネーム (ARN) を指定します。シークレットは、次のキーを持つ JSON 構造で格納されます。

    {
    "slackToken": "token"
}
    注記

    認証情報とシークレットは定期的に更新またはローテーションしてください。セキュリティに必要なアクセスレベルのみを指定してください。データソース、およびコネクタバージョン 1.0 と 2.0 (該当する場合) 間で認証情報とシークレットを再利用することはお勧めしません。

  • IAM ロール —RoleArnCreateDataSource コールするタイミングを指定して、IAMSecrets Managerシークレットにアクセスする権限をロールに付与し、Slack コネクタに必要なパブリック APIAmazon Kendra を呼び出すことができます。詳細については、Slack データソースの IAM ロールを参照してください。

オプションで、次の機能を追加することもできます。

  • Virtual Private Cloud (VPC) —VpcConfiguration データソース設定の一部として指定します。VPCAmazon Kendra を使用するように設定するを参照してください。

  • 変更ログ —Amazon Kendra インデックス内のドキュメントを追加、更新、削除する必要があるかどうかを判断するために、Slack データソースの変更ログメカニズムを使用するか。

    注記

    Amazon Kendraドキュメントをすべてスキャンしたくない場合は、変更ログを使用します。変更ログが大きい場合、Slack データソースのドキュメントをスキャンする方が、Amazon Kendra変更ログを処理するよりも時間が短くなる場合があります。Slack データソースをインデックスに初めて同期する場合は、すべてのドキュメントがスキャンされます。

  • 包含フィルターと除外フィルター — 特定のパブリックチャンネルとプライベートチャンネル、グループメッセージ、プライベートメッセージ、ボットメッセージとアーカイブメッセージを含めるか除外するかを指定します。Slack の認証情報の一部としてボットトークンを使用する場合は、インデックスに登録するチャンネルにボットトークンを追加する必要があります。ボットトークンを使用してダイレクトメッセージやグループメッセージをインデックスすることはできません。

    注記

    ほとんどのデータソースは正規表現パターンを使用します。正規表現パターンとは、フィルターと呼ばれる包含パターンまたは除外パターンです。包含フィルターを指定すると、包含フィルターと一致するコンテンツのみが索引付けされます。包含フィルターに一致しないドキュメントはインデックスに登録されません。包含フィルターと除外フィルターを指定した場合、除外フィルターに一致するドキュメントは、包含フィルターと一致してもインデックス作成されません。

  • フィールドマッピング — SlackAmazon Kendra データソースフィールドをインデックスフィールドにマッピングすることを選択します。詳細については、データソースフィールドのマッピングを参照してください。

  • ユーザーコンテキストフィルタリング — デフォルトで、データソースのアクセス制御リスト (ACL)Amazon Kendra をクロールします。ACL 情報は、ドキュメントへのユーザーまたはグループのアクセス権に基づいて検索結果をフィルタリングするために使用されます。詳しくは、「Slack データソースのユーザーコンテキストフィルター」を参照してください。
