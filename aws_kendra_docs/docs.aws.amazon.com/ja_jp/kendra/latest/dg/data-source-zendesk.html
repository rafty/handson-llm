Zendesk

Zendeskは、企業がカスタマーサポートとのやりとりを自動化および強化するのに役立つ顧客関係管理システムです。Zendeskサポートチケット、チケットコメント、チケット添付ファイル、ヘルプセンター記事、記事コメント、記事コメント添付ファイル、ガイドコミュニティトピック、コミュニティ投稿、コミュニティ投稿コメントのインデックスを作成できます。Amazon Kendra

特定の組織内のチケットのみにインデックスを付ける場合は、組織名でフィルタリングできます。Zendeskからのデータのクロールを開始するクロール日を設定することもできます。

Amazon KendraAmazon KendraTemplateConfigurationコンソールとAPIを使用してZendeskデータソースに接続できます。

サポートされている機能

Amazon KendraZendesk データソースコネクタは以下の機能をサポートしています。

  • 変更ログ

  • フィールドマッピング

  • ユーザーコンテキストのフィルタリング

  • 包含/除外フィルター

  • Virtual Private Cloud (VPC)

前提条件

Amazon Kendraを使用してZendeskデータソースのインデックスを作成する前に、AWS Zendeskとアカウントでこれらの変更を行います。

Zendeskでは、次のものが揃っていることを確認してください。

  • Zendesk Suite（プロフェッショナル/エンタープライズ）の管理者アカウントを作成しました。

  • ZendeskのホストURLを書き留めました。たとえば、https://{sub-domain (https://{host/)}.zendesk.com/)。

  • クライアント ID、クライアントシークレット、ユーザー名、およびパスワードを含む OAuth 2.0 認証トークンを生成しました。詳しくは、OAuth 2.0トークンの生成に関するZendeskのドキュメントを参照してください。

  • 次の OAuth 2.0 スコープが追加されました。

    • read

  • オプション:Amazon Kendra接続を許可する SSL 証明書をインストールしました。

  • 各ドキュメントが、Zendeskおよび同じインデックスに使用する予定の他のデータソース間で一意であることを確認しました。インデックスに使用する各データソースには、データソース間で同じドキュメントが含まれていてはなりません。ドキュメント ID はインデックス全体に適用され、インデックスごとに一意でなければなりません。

AWSアカウントには、次のものがあることを確認してください。

  • Amazon Kendraインデックスを作成し、API を使用している場合は、インデックス ID を記録しました。

  • IAMデータソースのロールを作成し、API を使用している場合は、IAMロールの ARN を記録しました。

  • AWSSecrets ManagerZendeskの認証情報をシークレットに保存し、APIを使用している場合はシークレットのARNを記録しました。

    注記

    認証情報とシークレットは定期的に更新またはローテーションしてください。セキュリティに必要なアクセスレベルのみを指定してください。データソース、およびコネクタバージョン 1.0 と 2.0 (該当する場合) 間で認証情報とシークレットを再利用することはお勧めしません。

IAM既存のロールまたはシークレットがない場合は、Zendeskデータソースをに接続するときに、IAMSecrets Managerコンソールを使用して新しいロールとシークレットを作成できますAmazon Kendra。API を使用している場合は、IAMSecrets Manager既存のロールとシークレットの ARN、およびインデックス ID を指定する必要があります。

接続手順

Amazon KendraZendeskデータソースに接続するには、Amazon Kendraデータにアクセスできるように、Zendeskデータソースの必要な詳細を入力する必要があります。Zendeskをまだ設定していない場合は、Amazon Kendraを参照してください前提条件。

Console

Amazon KendraZendeskに接続するには

  1. AWS マネジメントコンソールにサインインし、Amazon Kendra コンソールを開きます。

  2. 左側のナビゲーションペインで、[インデックス] を選択し、インデックスのリストから使用するインデックスを選択します。

    注記

    ユーザーアクセス制御設定は、インデックス設定で設定または編集できます。

  3. 「はじめに」ページで、「データソースを追加」を選択します。

  4. 「データソースの追加」ページで「Zendesk コネクタ」を選択し、「データソースを追加」を選択します。

  5. [データソースの詳細を指定] ページで、次の情報を入力します。

    1. [名前と説明] の [データソース名] に、データソースの名前を入力します。ハイフンは使用できますが、スペースは使用できません。

    2. (オプション) 説明 — データソースの説明を入力します。

    3. 「言語」(「デフォルト言語」)-索引用にドキュメントをフィルタリングする言語。特に指定しない限り、言語はデフォルトで英語に設定されます。メタデータで指定された言語は、選択した言語よりも優先されます。

    4. [タグ] の [新しいタグの追加] で、リソースを検索してフィルタリングしたり、AWSコストを追跡したりできます。

    5. [Next] (次へ) を選択します。

  6. 「アクセスとセキュリティの定義」ページで、次の情報を入力します。

    1. Zendesk URL — Zendesk URL を入力します。

    2. AWSSecrets Managersecret —既存のシークレットを選択するか、Secrets Manager Zendesk認証情報を保存する新しいシークレットを作成します。新しいシークレットを作成することを選択すると、AWSSecrets Managerシークレットウィンドウが開きます。

      1. 「AWSSecrets Managerシークレットの作成」ウィンドウに次の情報を入力します。

        1. シークレット名 — シークレットの名前。シークレット名には「AmazonKendra-Zendesk-」というプレフィックスが自動的に追加されます。

        2. クライアントID、クライアントシークレット、ユーザー名、パスワードには、Zendeskアカウントで作成した認証資格値を入力します。

      2. [Save] (保存) を選択します。

    3. Virtual Private Cloud (VPC) — VPC の使用を選択できます。その場合は、サブネットと VPC セキュリティグループを追加する必要があります。

    4. IAM ロール — 既存の IAM ロールを選択するか、リポジトリの認証情報にアクセスしてコンテンツをインデックス作成します。

      注記

      インデックスに使用される IAM ロールはデータソースには使用できません。既存のロールがインデックスや FAQ に使用されているかどうかわからない場合は、エラーを避けるため [新しいロールを作成する] を選択してください。

    5. [Next] (次へ) を選択します。

  7. [同期設定の設定] ページで、次の情報を入力します。

    1. エンティティまたはコンテンツタイプの選択 — クロールするZendeskのエンティティまたはコンテンツタイプ。

    2. 変更ログ — 選択すると、すべてのファイルを同期する代わりにインデックスを更新できます。

    3. 組織名 — Zendeskの組織名を入力して同期をフィルタリングします。

    4. 同期開始日 — コンテンツのインデックス作成を開始する日付。

    5. Regex パターン — 特定のファイルを包含パターンまたは除外する正規表現パターン。最大 100 個のパターンを追加できます。

    6. 頻度の同期実行スケジュール —Amazon Kendra データソースと同期する頻度を選択します。

    7. [Next] (次へ) を選択します。

  8. [フィールドマッピングの設定] ページで、次の情報を入力します。

    1. チケット、チケットコメント、チケットコメント添付ファイル、記事、記事コメント、記事コメント添付、コミュニティトピック、コミュニティ投稿、コミュニティ投稿コメントの場合 —Amazon Kendra 生成されたデフォルトデータソースフィールドから、インデックスにマッピングしたいものを選択します。

    2. フィールドを追加 — カスタムデータソースフィールドを追加して、マッピング先のインデックスフィールド名とフィールドデータタイプを作成します。

    3. [Next] (次へ) を選択します。

  9. [確認と作成] ページで、入力した情報が正しいことを確認し、[データソースの追加] を選択します。また、このページで情報を編集することもできます。データソースが正常に追加されると、データソースページに表示されます。

API

Amazon KendraZendeskに接続するには

TemplateConfigurationAPI を使用してデータソーススキーマの JSON を指定する必要があります。これには、以下の情報を入力する必要があります。

  • データソース-データソースをとして指定しますZENDESK。

  • ホストURL — 接続設定またはリポジトリエンドポイントの詳細の一部として、ZendeskホストのURLを指定します。たとえば、https://yoursubdomain.zendesk.com。

  • タイプ —TEMPLATE 電話をかけるときにタイプとして指定しますCreateDataSource。

  • シークレットの Amazon リソースネーム (ARN) — ZendeskSecrets Manager アカウントの認証情報を含むシークレットの Amazon リソースネーム (ARN) を指定します。シークレットは、次のキーを持つ JSON 構造に保存されます。

    {
    "hostUrl": "https://yoursubdomain.zendesk.com",
    "clientId": "kendra",
    "clientSecret": "Zendesk client secret",
    "userName": "Zendesk user name",
    "password": "Zendesk password"
}
    注記

    認証情報とシークレットは定期的に更新またはローテーションしてください。セキュリティに必要なアクセスレベルのみを指定してください。データソース、およびコネクタバージョン 1.0 と 2.0 (該当する場合) 間で認証情報とシークレットを再利用することはお勧めしません。

  • IAMロール —RoleArnCreateDataSource いつ呼び出すかを指定して、Secrets Managerシークレットにアクセスしたり、ZendeskIAMAmazon Kendra Connectorに必要なパブリックAPIを呼び出したりする権限をロールに与えます。詳細については、「Zendesk データソースの IAM ロール」を参照してください。

オプションで、次の機能を追加することもできます。

  • [Virtual Private Cloud (VPC)] —VpcConfiguration 電話をかけるときに指定しますCreateDataSource。VPCAmazon Kendra を使用するように設定する を参照してください。

  • 変更ログ —Amazon Kendra インデックス内のドキュメントを追加、更新、削除する必要があるかどうかを判断するために、Zendeskデータソースの変更ログメカニズムを使用するか。

    注記

    Amazon Kendraドキュメントをすべてスキャンしたくない場合は、変更ログを使用します。変更ログが大きい場合、Zendesk データソース内のドキュメントは、Amazon Kendra変更ログを処理するよりも短くなる場合があります。Zendeskデータソースをインデックスに初めて同期する場合は、すべてのドキュメントがスキャンされます。

  • 包含フィルターと除外フィルター — 以下を含めるか除外するかを指定します。

    • Support チケット、チケットコメント、チケットコメント添付ファイル

    • ヘルプセンターの記事、記事の添付ファイル、記事のコメント

    • コミュニティのトピック、投稿をガイドしたり、コメントを投稿したりできます

  • 注記

    ほとんどのデータソースは正規表現パターンを使用します。正規表現パターンとは、フィルターと呼ばれる包含パターンまたは除外パターンです。包含フィルターを指定すると、包含フィルターと一致するコンテンツのみが索引付けされます。包含フィルターに一致しないドキュメントはインデックスに登録されません。包含パターンと除外フィルターを指定した場合、除外フィルターに一致するドキュメントは、包含フィルターと一致してもインデックス作成されません。

  • フィールドマッピング —Amazon Kendra Zendeskのデータソースフィールドをインデックスフィールドにマッピングすることを選択します。詳細については、データソースフィールドのマッピングを参照してください。

  • ユーザーコンテキストフィルタリング — デフォルトで、データソースのアクセス制御リスト (ACL)Amazon Kendra をクロールします。ACL 情報は、ドキュメントへのユーザーまたはグループのアクセス権に基づいて検索結果をフィルタリングするために使用されます。詳しくは、「Zendeskデータソースのユーザーコンテキストフィルタリング」を参照してください。

設定するその他の重要な JSON キーのリストについては、を参照してくださいZendesk テンプレートスキーマ。
