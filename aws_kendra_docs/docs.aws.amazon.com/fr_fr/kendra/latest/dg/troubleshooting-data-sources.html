Résolution des problèmes de sources de données

Cette section peut vous aider à résoudre les problèmes liés aux sources de données Amazon Kendra.

Mes documents n'ont pas été indexés

Lorsque vous synchronisez votre index Kendra avec une source de données, vous pouvez rencontrer des problèmes qui empêchent l'indexation des documents. L'indexation est un processus en deux étapes. Tout d'abord, la source de données est vérifiée pour détecter les documents nouveaux et mis à jour à indexer, et pour rechercher les documents à supprimer de l'index. Ensuite, au niveau du document, chaque document est consulté et indexé.

Une erreur peut survenir au cours de l'une ou l'autre de ces étapes. Les erreurs au niveau de la source de données sont signalées dans la console dans la section Historique des exécutions de la page de détails de la source de données. Le statut de la tâche de synchronisation peut être Réussi, Inachevé ou Échec. Vous pouvez également voir le nombre de documents indexés et supprimés au cours de la tâche. Si le statut est Échec, un message s'affiche dans la colonne Détails.

Les erreurs au niveau du document sont signalées dans Amazon CloudWatch Logs. Vous pouvez voir les erreurs à l'aide de la CloudWatch console.

Ma tâche de synchronisation a échoué

Une tâche de synchronisation échoue généralement en cas d'erreur de configuration dans l'index ou la source de données. Le message d'erreur figurant dans la colonne Détails de la source de données fournit des informations sur le problème. Le problème est généralement dû au fait que l'index ou la source de données ne dispose pas des autorisations IAM appropriées. Le message d'erreur décrit les autorisations manquantes. Voici certains des messages d'erreur que vous pouvez recevoir :

Failed to create log group for job. Please make sure that the IAM role provided has sufficient permissions.

Si vous n'êtes pas autorisé à utiliser votre rôle d'index CloudWatch, la source de données ne pourra pas créer de CloudWatch journal. Si cette erreur s'affiche, vous devez ajouter CloudWatch des autorisations au rôle d'index.

Failed to access S3 file prefix (bucket name) while trying to crawl your metadata files. Please make sure the IAM Role (role ARN) provided has sufficient permissions.

Lorsque vous utilisez une source de données Amazon S3, Kendra doit être autorisée à accéder au compartiment contenant les documents. Vous devez ajouter l'autorisation à Kendra de lire le compartiment dans le rôle IAM de la source de données.

The provided IAM Role (role ARN) could not be assumed. Please make sure Amazon Kendra is a trusted entity that is allowed to assume the role.

Kendra a besoin d'une autorisation pour assumer les rôles IAM d'index et de source de données. Vous devez ajouter une politique de confiance aux rôles avec autorisation pour l'sts:AssumeRoleaction.

Pour connaître les politiques IAM dont Kendra a besoin pour indexer une source de données, consultezIAMrôles d'accès pourAmazon Kendra.

Ma tâche de synchronisation est incomplète

Les tâches sont généralement incomplètes lorsqu'elles ont terminé le processus au niveau de la source de données, mais comportent des erreurs au cours du processus au niveau du document. Lorsqu'une tâche est incomplète, il se peut que certains documents aient tout de même été indexés avec succès. Pour une source de données Amazon S3, une tâche incomplète est généralement due à :

  • Les métadonnées d'un ou de plusieurs documents n'étaient pas valides.

  • Lorsque des documents sont soumis pour indexation mais qu'au moins un document n'a pas été soumis.

  • Lorsque des documents sont soumis pour être supprimés de l'index mais qu'au moins un document n'a pas été soumis.

Pour résoudre les problèmes liés à une tâche de synchronisation incomplète, consultez d'abord vos CloudWatch journaux.

  1. Dans la colonne des détails, choisissez Afficher les détails dans CloudWatch.

  2. Consultez les messages d'erreur pour déterminer la cause de l'échec du document.

Ma tâche de synchronisation a réussi, mais aucun document n'est indexé

Parfois, l'exécution d'une tâche de synchronisation d'index est marquée comme réussie, mais aucun document nouveau ou mis à jour n'est indexé à la date prévue. Les raisons possibles incluent :

  • Vérifiez la CloudWatch DocumentsSubmittedForIndexingFailed métrique pour voir si des documents n'ont pas pu être synchronisés. Consultez vos CloudWatch journaux pour plus de détails.

  • Pour une source de données Amazon S3, vous avez peut-être donné à Kendra un nom de compartiment ou un préfixe erroné. Assurez-vous que le compartiment utilisé par Kendra est celui qui contient les documents à indexer.

  • Lors de la réindexation d'un document qui n'a pas pu être indexé dans une tâche précédente, Kendra ne l'indexe que si vous avez modifié le document ou le fichier de métadonnées associé.

Je rencontre des problèmes de format de fichier lors de la synchronisation de ma source de données

Si vous rencontrez des problèmes de format de fichier lors de l'ajout de fichiers à votre source de données ou de la synchronisation de votre source de données, assurez-vous que vos types de documents sont pris en charge par Kendra. Pour obtenir la liste des types de documents pris en charge par,Amazon Kendra voir Types de documents.

Si vous utilisez l'BatchPutDocumentAPI avec des fichiers texte brut, spécifiezPLAIN_TEXT le type de contenu.

Combien de temps prend la synchronisation d'une source de données ?

Si aucune mise à jour n'est apportée aux documents, le temps de synchronisation d'un index Amazon Kendra augmente proportionnellement au nombre de documents. Par exemple, la synchronisation de 1 000 documents sans mise à jour prendra environ cinq minutes et la synchronisation de 2 000 documents sans mise à jour prendra environ 10 minutes. En cas de mise à jour des documents, le temps de synchronisation augmentera en fonction du nombre de documents mis à jour.

Quels sont les frais de synchronisation d'une source de données ?

Lorsque vous synchronisez votre index, il faut deux minutes pour le préchauffer et activer Amazon EC2 afin d'établir les connexions nécessaires. Vous n'êtes pas débité au cours de ce processus. Votre compteur d'utilisation ne démarre qu'après le démarrage de la tâche de synchronisation. Pour en savoir plus sur les tarifs de Kendra, consultez les tarifs d'Amazon Kendra.

Je reçois une erreur d'autorisation Amazon EC2

Si une erreur de fonctionnement non autorisée Amazon EC2 se produit lors d'une synchronisation pour une source de données de cloud privé virtuel (VPC), il est probable que votre rôle IAM VPC ne dispose pas des autorisations requises. Vérifiez que le rôle IAM que vous utilisez pour votre source de données possède les autorisations associées. Pour plus d'informations, veuillez consulter IAMRôle de Virtual (VPC).

Je ne parviens pas à utiliser les liens d'index de recherche pour ouvrir mes objets Amazon S3

Votre index Kendra ne peut accéder qu'aux fichiers auxquels une source de données Amazon S3 lui accorde des autorisations d'accès. Par exemple, Kendra ne peut pas modifier les autorisations Amazon S3 qui déterminent si un objet est censé être public ou crypté. Kendra ne dispose pas non plus des autorisations par défaut pour créer ou renvoyer un lien signé pour les objets Amazon S3. Si vous souhaitez activer les liens signés pour les objets Amazon S3 dans un index Kendra, deux options s'offrent à vous :

  • Vous pouvez signer les résultats de votre requête d'index avec l'objet uri source avant de renvoyer le résultat à la page de recherche. Pour une step-by-step présentation détaillée de ce processus, consultez la section Partage d'objets à l'aide d'URL présignées.

  • Vous pouvez remplacer l'URI source des métadonnées des objets Amazon S3 et rendre votre service disponible via un réseau de diffusion de CloudFront contenu (CDN) connecté à un compartiment Amazon S3. Vous pouvez également utiliser un point de terminaison de proxy API Gateway qui renvoie une URL présignée et redirige vers celle-ci.

Je reçois un message d'erreur AccessDenied lors de l'utilisation d'un fichier de certificat SSL

Si vous recevez un message d'erreur d'accès refusé lorsque vous utilisez un certificat SSL avec votre source de données, assurez-vous que votre rôle IAM est autorisé à accéder au fichier de certificat SSL à l'emplacement spécifié. Si le certificat est chiffré à l'aide d'uneAWS KMS clé, votre rôle IAM doit également être autorisé à le déchiffrer à l'aide de cetteAWS KMS clé. Pour de plus amples informations, veuillez consulter Authentification et contrôle d'accès pour leAWS KMS.

Je reçois une erreur d'autorisation lors de l'utilisation d'une source SharePoint de données

Si une erreur d'autorisation s'affiche lors de la synchronisation de votre index avec une source de SharePoint données, vérifiez qu'un rôle d'administrateur de site vous a été attribué dans SharePoint.

Mon index n'explore pas les documents de ma source de données Confluence

Si votre index Kendra n'explore pas les documents de votre source de données Confluence pendant le processus de synchronisation, vérifiez que vous faites partie des groupes d'administrateurs de Confluence.
