Connecteur de source de données personnalisé

Utilisez une source de données personnalisée lorsque vous disposez d'un référentiel quiAmazon Kendra ne fournit pas encore de connecteur de source de données pour. Vous pouvez l'utiliser pour consulter les mêmes mesures d'historique des exécutions que celles fournies par les sources deAmazon Kendra données, même lorsque vous ne pouvez pas les utiliserAmazon Kendra pour synchroniser vos référentiels. Utilisez-le pour créer une expérience de surveillance de la synchronisation cohérente entre les sources deAmazon Kendra données et les sources personnalisées. Plus précisément, utilisez une source de données personnalisée pour voir les métriques de synchronisation d'un connecteur de source de données que vous avez créé à l'aide BatchDeleteDocumentdes API BatchPutDocumentet.

Pour résoudre votre connecteur de source de données Amazon Kendra, consultezRésolution des problèmes de sources de données.

Lorsque vous créez une source de données personnalisée, vous avez un contrôle total sur la manière dont les documents à indexer sont sélectionnés. Vous devez spécifier le titre principal de vos documents à l'aide de l'objet Document, afin d'avoirDocumentTitle et d'DocumentURIinclure dans la réponse leQuery résultat._source_uri DocumentAttribute

Vous créez un identifiant pour votre source de données personnalisée à l'aide de la console ou de l'CreateDataSourceAPI. Pour utiliser la console, attribuez un nom à votre source de données, et éventuellement une description et des balises de ressource. Une fois la source de données créée, un identifiant de source de données s'affiche. Copiez cet ID à utiliser lors de la synchronisation de la source de données avec l'index.

Vous pouvez également créer une source de données personnalisée à l'aide de l'CreateDataSourceAPI. L'API renvoie un identifiant à utiliser lors de la synchronisation de la source de données. Lorsque vous utilisez l'CreateDataSourceAPI pour créer une source de données personnalisée, vous ne pouvez pas définirConfiguration lesSchedule paramètresRoleArn ou. Si vous définissez ces paramètres,Amazon Kendra renvoie uneValidationException exception.

Pour utiliser une source de données personnalisée, créez une application chargée de mettre à jour l'Amazon Kendraindex. L'application dépend du robot d'exploration que vous créez. Le robot d'exploration lit les documents de votre référentiel et détermine à qui ils doivent être envoyésAmazon Kendra. Votre application doit effectuer les étapes suivantes :

  1. Parcourez votre référentiel et dressez une liste des documents qui y ont été ajoutés, mis à jour ou supprimés.

  2. Appelez l'StartDataSourceSyncJobAPI pour signaler qu'une tâche de synchronisation est en cours de démarrage. Vous fournissez un ID de source de données pour identifier la source de données en cours de synchronisation. Amazon Kendrarenvoie un ID d'exécution pour identifier une tâche de synchronisation particulière.

  3. Appelez l'BatchDeleteDocumentAPI pour supprimer des documents de l'index. Vous fournissez l'ID de source de données et l'ID d'exécution pour identifier la source de données en cours de synchronisation et la tâche à laquelle cette mise à jour est associée.

  4. Appelez l'StopDataSourceSyncJobAPI pour signaler la fin de la tâche de synchronisation. Après avoir appelé l'StopDataSourceSyncJobAPI, l'ID d'exécution associé n'est plus valide.

  5. Appelez l'ListDataSourceSyncJobsAPI à l'aide de l'index et des identifiants de source de données pour répertorier les tâches de synchronisation pour la source de données et pour voir les métriques associées aux tâches de synchronisation.

Après avoir terminé une tâche de synchronisation, vous pouvez en démarrer une nouvelle. Il peut s'écouler un certain temps avant que tous les documents soumis ne soient ajoutés à l'index. Utilisez l'ListDataSourceSyncJobsAPI pour voir l'état de la tâche de synchronisation. Si leStatus résultat de la tâche de synchronisation est renvoyéSYNCING_INDEXING, certains documents sont toujours en cours d'indexation. Vous pouvez démarrer une nouvelle tâche de synchronisation lorsque le statut de la tâche précédente estFAILEDSUCCEEDED, ouSYNCING_INDEX.

Après avoir appelé l'StopDataSourceSyncJobAPI, vous ne pouvez pas utiliser d'identifiant de tâche de synchronisation dans un appel auxBatchDeleteDocument APIBatchPutDocument or. Dans ce cas, tous les documents soumis sont renvoyés dans le message deFailedDocuments réponse de l'API.

Attributs requis

Lorsque vous soumettez un document àAmazon Kendra l'aide de l'BatchPutDocumentAPI, chaque document nécessite deux attributs pour identifier la source de données et l'exécution de synchronisation auxquelles il appartient. Vous devez fournir les deux attributs suivants :

  • _data_source_id—Identificateur de la source de données. Cela est renvoyé lorsque vous créez la source de données à l'aide de la console ou de l'CreateDataSourceAPI.

  • _data_source_sync_job_execution_id—L'identifiant de l'exécution de synchronisation. Ceci est renvoyé lorsque vous démarrez la synchronisation de l'index avec l'StartDataSourceSyncJobAPI.

Voici le code JSON requis pour indexer un document à l'aide d'une source de données personnalisée.

{
    "Documents": [
        {
            "Attributes": [
                {
                    "Key": "_data_source_id",
                    "Value": {
                        "StringValue": "data source identifier"
                    }
                },
                {
                    "Key": "_data_source_sync_job_execution_id",
                    "Value": {
                        "StringValue": "sync job identifier"
                    }
                }
            ],
            "Blob": "document content",
            "ContentType": "content type",
            "Id": "document identifier",
            "Title": "document title"
        }
    ],
    "IndexId": "index identifier",
    "RoleArn": "IAM role ARN"
}

Lorsque vous supprimez un document de l'index à l'aide de l'BatchDeleteDocumentAPI, vous devez spécifier les deux champs suivants dans leDataSourceSyncJobMetricTarget paramètre :

  • DataSourceId—Identificateur de la source de données. Cela est renvoyé lorsque vous créez la source de données à l'aide de la console ou de l'CreateDataSourceAPI.

  • DataSourceSyncJobId—L'identifiant de l'exécution de synchronisation. Ceci est renvoyé lorsque vous démarrez la synchronisation de l'index avec l'StartDataSourceSyncJobAPI.

Voici le code JSON requis pour supprimer un document de l'index à l'aide de l'BatchDeleteDocumentAPI.

{
    "DataSourceSyncJobMetricTarget": {
        "DataSourceId": "data source identifier",
        "DataSourceSyncJobId": "sync job identifier"
    },
    "DocumentIdList": [
        "document identifier"
    ],
    "IndexId": "index identifier"
}

Affichage des métriques

Une fois qu'une tâche de synchronisation est terminée, vous pouvez utiliser l'DataSourceSyncJobMetricsAPI pour obtenir les mesures associées à la tâche de synchronisation. Utilisez-le pour surveiller les synchronisations de vos sources de données personnalisées.

Si vous soumettez le même document plusieurs fois, soit dans le cadre de l'BatchPutDocumentAPI, soit de l'BatchDeleteDocumentAPI, soit s'il est soumis à la fois pour ajout et suppression, le document n'est compté qu'une seule fois dans les statistiques.

  • DocumentsAdded—Le nombre de documents soumis à l'aide de l'BatchPutDocumentAPI associée à cette tâche de synchronisation ajoutée à l'index pour la première fois. Si un document est soumis pour ajout plusieurs fois lors d'une synchronisation, le document n'est compté qu'une seule fois dans les statistiques.

  • DocumentsDeleted—Le nombre de documents soumis à l'aide de l'BatchDeleteDocumentAPI associée à cette tâche de synchronisation a été supprimé de l'index. Si un document est soumis pour suppression plusieurs fois lors d'une synchronisation, le document n'est compté qu'une seule fois dans les statistiques.

  • DocumentsFailed—Le nombre de documents associés à cette tâche de synchronisation dont l'indexation a échoué. Il s'agit de documents qui ont été acceptésAmazon Kendra pour l'indexation mais qui n'ont pas pu être indexés ou supprimés. Si un document n'est pas accepté parAmazon Kendra, l'identifiant du document est renvoyé dans la propriété deFailedDocuments réponse desBatchDeleteDocument APIBatchPutDocument and.

  • DocumentsModified—Le nombre de documents modifiés soumis à l'aide de l'BatchPutDocumentAPI associée à cette tâche de synchronisation qui ont été modifiés dans l'Amazon Kendraindex.

Amazon Kendraémet égalementAmazon CloudWatch des métriques lors de l'indexation de documents. Pour plus d'informations, consultez surveillerAmazon Kendra unAmazon CloudWatch.

Amazon Kendrane renvoie pas laDocumentsScanned métrique pour les sources de données personnalisées. Il émet également lesCloudWatch métriques répertoriées dans le document Métriques pour les sources deAmazon Kendra données.
