Comportement de nouvelle tentative

Le comportement de nouvelle tentative inclut des paramètres concernant la manière dont les SDK tentent de se rétablir après des échecs résultant de demandes adressées àServices AWS.

Configurez cette fonctionnalité à l'aide des éléments suivants :

max_attempts- paramètre deAWSconfig fichier partagé
AWS_MAX_ATTEMPTS- variable d'environnement

Spécifie le nombre maximal de tentatives à effectuer sur une demande.

Valeur par défaut : si cette valeur n'est pas spécifiée, sa valeur par défaut dépend de la valeur duretry_mode paramètre :

  • Siretry_mode c'est le caslegacy : utilise une valeur par défaut spécifique à votre SDK (consultez le guide de votre SDK spécifique ou la base de code de votre SDK pour connaître lamax_attempts valeur par défaut).

  • Siretry_modestandard c'est le cas, effectue trois tentatives.

  • Siretry_modeadaptive c'est le cas, effectue trois tentatives.

Valeurs valides : nombre supérieur à 0.

retry_mode- paramètre deAWSconfig fichier partagé
AWS_RETRY_MODE- variable d'environnement

Spécifie la manière dont le SDK ou l'outil de développement tente de recommencer.

Valeur par défaut :legacy il s'agit de la stratégie de nouvelle tentative par défaut.

Valeurs valides:

  • legacy— Spécifique à votre SDK (consultez le guide de votre SDK spécifique ou la base de code de votre SDK).

  • standard— Ensemble standard de règles relatives aux nouvelles tentatives dans lesAWS SDK. Ce mode inclut un ensemble standard d'erreurs qui sont réessayées et prend en charge les quotas de nouvelles tentatives. Le nombre maximum de tentatives par défaut avec ce mode est de trois, saufmax_attempts configuration explicite.

  • adaptive— Un mode de nouvelle tentative expérimental qui inclut les fonctionnalités du mode standard mais inclut une limitation automatique côté client. Ce mode étant expérimental, il est possible qu'il modifie le comportement à l'future.

Voici le pseudocode de haut niveau pour le modestandard et le modeadaptive nouvelle tentative :

MakeSDKRequest() {
  attempts = 0
  loop {
    GetSendToken()
    response = SendHTTPRequest()
    RequestBookkeeping(response)
    if not Retryable(response)
      return response
    attempts += 1
    if attempts >= MAX_ATTEMPTS:
      return response
    if not HasRetryQuota(response)
      return response
    delay = ExponentialBackoff(attempts)
    sleep(delay)
  }
}

Voici plus de détails sur les composants utilisés dans le pseudocode :

GetSendToken:

Les compartiments à jetons ne sont utilisés qu'en modeadaptive nouvelle tentative. Les compartiments de jetons imposent un taux de demandes maximal en exigeant qu'un jeton soit disponible pour lancer une demande. Le client SDK est configurable pour échouer rapidement à la demande ou pour la bloquer jusqu'à ce qu'un jeton soit disponible.

La limitation du débit côté client est un algorithme qui permet initialement de faire des demandes à n'importe quel rythme, dans la limite des jetons autorisés. Cependant, après la détection d'une réponse limitée, le client rate-of-request est alors limité en conséquence. L'allocation de jetons est également augmentée en conséquence si des réponses positives sont reçues.

Grâce à la limitation de débit adaptative, les SDK peuvent ralentir la vitesse à laquelle les demandes sont envoyées afin de mieux s'adapter à la capacité deServices AWS.

SendHTTPRequest:

La plupartAWS des SDK utilisent une bibliothèque HTTP qui utilise des pools de connexions afin que vous puissiez réutiliser une connexion existante lorsque vous envoyez une requête HTTP. En général, les connexions sont réutilisées lors de nouvelles tentatives de requêtes en raison d'erreurs de limitation. Les demandes ne sont pas réutilisées lors d'une nouvelle tentative en raison d'erreurs passagères.

RequestBookkeeping:

Le quota de nouvelles tentatives doit être mis à jour si la demande aboutit. Pour le modeadaptive nouvelle tentative uniquement, la variable d'étatmaxsendrate est mise à jour en fonction du type de réponse reçue.

Retryable:

Cette étape détermine si une réponse peut être réessayée en fonction des critères suivants :

  • Le code d'état HTTP .

  • Code d'erreur renvoyé par le service.

  • Erreurs de connexion, définies comme toute erreur reçue par le SDK dans laquelle aucune réponse HTTP du service n'est reçue.

Les erreurs transitoires (codes d'état HTTP 400, 408, 500, 502, 503 et 504) et les erreurs de limitation (codes d'état HTTP 400, 403, 429, 502, 503 et 509) peuvent toutes être réessayées. Le comportement des nouvelles tentatives du SDK est déterminé en combinaison avec des codes d'erreur ou d'autres données provenant du service.

MAX_ATTEMPTS:

Spécifié par le paramètreconfig du fichier ou la variable d'environnement.

HasRetryQuota

Cette étape limite les demandes de nouvelle tentative en exigeant qu'un jeton soit disponible dans le compartiment de quotas de nouvelle tentative. Les compartiments de quotas de nouvelle tentative sont un mécanisme destiné à empêcher les nouvelles tentatives qui ont peu de chances d'aboutir. Ces quotas dépendent du SDK, dépendent souvent du client et parfois même des points de terminaison du service. Les jetons de quota disponibles pour les nouvelles tentatives sont supprimés lorsque les demandes échouent pour diverses raisons, et réapprovisionnés lorsqu'elles aboutissent. Lorsqu'il ne reste aucun jeton, la boucle de nouvelle tentative est terminée.

ExponentialBackoff

Pour une erreur qui peut être réessayée, le délai de nouvelle tentative est calculé à l'aide d'un délai exponentiel tronqué. Les SDK utilisent un backoff exponentiel binaire tronqué avec jitter. L'algorithme suivant montre comment le temps de sommeil, en secondes, est défini pour une réponse à une demandei : 

seconds_to_sleep_i = min(b*r^i, MAX_BACKOFF)

Dans l'algorithme précédent, les valeurs suivantes s'appliquent :

b = random number within the range of: 0 <= b <= 1

r = 2

MAX_BACKOFF = 20 secondspour la plupart des SDK. Consultez le guide ou le code source de votre SDK spécifique pour obtenir une confirmation.
